Chapter 2

Understanding Firecracker MicroVMs

How Firecracker achieves VM-level isolation with container-like speed through a minimal device model and KVM-based hardware virtualization.

What Makes Firecracker Different from Docker and Traditional VMs

To understand where Firecracker sits, you need to understand the three dominant isolation strategies and the tradeoffs each makes between security, performance, and density.

Process / Container Isolation (Docker)

Docker containers share the host Linux kernel. Isolation comes from kernel namespaces (PID, network, mount, user) and cgroups for resource limits. The container runtime never boots an OS — it calls clone() with namespace flags and drops into a chroot-like filesystem view.

This is fast (sub-second startup) and memory-efficient (tens of MiB overhead), but the shared kernel is the problem. A kernel vulnerability gives an attacker access to every container on the host. For multi-tenant workloads like AI coding agents running untrusted code, that is an unacceptable risk.

Traditional VMs (QEMU/KVM)

A traditional virtual machine manager like QEMU boots an entire guest OS with its own kernel. QEMU emulates dozens of devices: USB controllers, PCI bridges, sound cards, display adapters, ACPI tables. Each emulated device is C code that parses untrusted guest input — a large attack surface.

You get strong isolation (separate kernel, hardware-enforced memory boundaries), but at a cost: 200–500 MiB of overhead per VM, 3–10 seconds to boot, and QEMU itself is roughly 1.4 million lines of code.

Firecracker: The Middle Path

Firecracker (35.3k stars on GitHub) is a Virtual Machine Monitor (VMM) written in Rust. It gives you the same hardware-enforced isolation as QEMU/KVM — each guest has its own kernel, its own address space, separated by CPU privilege levels — but strips away everything a cloud workload does not need.

The result:

  • Startup in under 125 milliseconds
  • Memory overhead under 5 MiB per microVM
  • Creation rate of up to 150 microVMs per second on a single host
  • Only 5 emulated devices (vs. dozens in QEMU)

Under the hood: Firecracker is not a fork of QEMU. It was written from scratch in Rust specifically for serverless and container workloads. The entire codebase is roughly 50k lines of Rust — small enough for meaningful security audits.

The Five-Device Model: Minimal Attack Surface

Every emulated device in a VMM is code that processes input from the guest. A bug in any device emulation is a potential guest-to-host escape. QEMU emulates IDE controllers, floppy drives, VGA adapters, USB hubs, and more — most of which a serverless function never touches.

Firecracker exposes exactly five devices:

  1. virtio-net — Network interface. TAP-backed, high-performance paravirtualized networking.
  2. virtio-block — Block storage. Backed by a file on the host; the guest sees a disk.
  3. virtio-vsock — Host-guest communication channel without requiring network configuration. Used for control-plane RPCs between the Lambda runtime and the sandbox.
  4. Serial console — Text I/O for boot logs and debugging. Minimal parsing surface.
  5. i8042 keyboard controller — Used only to signal system reset (Ctrl+Alt+Del equivalent). No actual keyboard input is processed.

That is it. No USB, no PCI bridge, no display, no sound, no ACPI beyond the basics. Each device that does not exist is an entire class of vulnerabilities that cannot happen.

Production tip: The virtio devices use paravirtualization — the guest kernel knows it is in a VM and cooperates with a simple, well-defined protocol. This is both faster and simpler to audit than emulating real hardware behavior that was never designed for security.

KVM-Based Hardware Virtualization Explained for Backend Developers

If you have never worked below the application layer, CPU privilege levels can feel abstract. Here is how it works concretely.

CPU Ring Levels

x86 (and Arm) CPUs have privilege levels, historically called "rings":

  • Ring 0 — The OS kernel runs here. Full access to hardware, memory mapping, interrupts.
  • Ring 3 — User applications run here. Cannot touch hardware directly; must ask the kernel via syscalls.

Before hardware virtualization, running a guest OS was tricky. The guest kernel expects to be in Ring 0, but you cannot give it real Ring 0 access without losing control. Solutions like binary translation (VMware) or paravirtualization (Xen) worked but were slow or required modifying the guest.

Hardware Virtualization Extensions (VT-x / AMD-V)

Intel VT-x and AMD-V added a new privilege level below Ring 0, sometimes called "Ring -1" or VMX root mode. The hypervisor runs in VMX root mode. The guest kernel runs in Ring 0 as it expects, but the CPU itself traps sensitive operations (page table changes, I/O port access, MSR writes) to the hypervisor.

This means:

  • The guest kernel runs unmodified at near-native speed
  • The CPU hardware enforces that the guest cannot access host memory
  • Context switches between guest and host ("VM exits") are handled in hardware

What KVM Does

KVM (Kernel-based Virtual Machine) is a Linux kernel module that exposes VT-x/AMD-V to userspace via /dev/kvm. A userspace program (the VMM — in our case, Firecracker) does the following:

  1. Opens /dev/kvm and creates a VM file descriptor
  2. Allocates memory regions and maps them into the VM
  3. Creates vCPU file descriptors (one per virtual processor)
  4. Loads the guest kernel into the mapped memory
  5. Enters a loop: calls ioctl(KVM_RUN) to execute the guest, handles VM exits when the guest does something that needs emulation

The critical point: the guest code runs directly on the physical CPU. KVM is not interpreting instructions. When the guest does arithmetic, calls functions, or accesses its own memory, those operations happen at native speed. KVM only intervenes for privileged operations like I/O.

Under the hood: A "VM exit" happens when the guest tries to access a device port, execute a CPUID instruction, or touch memory that is not mapped. The CPU traps to KVM, which routes the exit to the userspace VMM (Firecracker). Firecracker handles it (e.g., processes a virtio-net packet) and resumes the guest. Minimizing these exits is key to performance — and having only 5 devices means fewer reasons for the guest to exit.

Security Layers in Firecracker

KVM provides hardware isolation, but Firecracker adds defense in depth:

  • Seccomp filters — Thread-specific seccomp-BPF profiles restrict which syscalls the VMM process can make. Even if an attacker escapes the VM, the process cannot call dangerous syscalls.
  • Jailer process — Before the VMM starts, a jailer sets up:
    • A chroot jail via pivot_root()
    • cgroups (v1 or v2) for CPU and memory limits
    • setrlimit() to cap file descriptors and other resources
    • Dedicated non-privileged UID/GID
    • Separate mount, PID, and network namespaces

Production hardening checklist:

  • Disable Simultaneous Multi-Threading (SMT) to prevent side-channel attacks between threads sharing a core
  • Disable Kernel Same-page Merging (KSM) to prevent memory deduplication side channels
  • Block Instance Metadata Service (IMDS) access from within VMs
  • Use DDR4 memory with Target Row Refresh (TRR) and ECC to mitigate Rowhammer attacks

Supported Hardware

Firecracker supports:

  • Intel: Cascade Lake through Granite Rapids
  • AMD: Milan and Genoa
  • Arm: Graviton 2 through Graviton 4

Guest OS support: Linux kernel 4.14+, Amazon Linux 2/2023, and Ubuntu 24.04.

Performance: <125ms Startup, <5 MiB Overhead, 150 VMs/sec

These are not theoretical numbers. They come from the Firecracker documentation and represent measured performance on supported hardware.

Why It Is This Fast

Traditional VM boot is slow because QEMU emulates BIOS/UEFI firmware, probes devices, initializes PCI buses, and loads bootloaders. Firecracker skips all of that:

  • No BIOS or UEFI. Firecracker loads the kernel directly into memory and sets up boot parameters itself.
  • No device probing. The guest kernel is compiled to expect exactly the 5 virtio devices.
  • No bootloader chain. The kernel starts executing immediately.
  • Minimal memory allocation. Only the 5 MiB VMM overhead plus whatever you allocate to the guest.

What 150 VMs/Second Means in Practice

On a single host, you can spin up 150 new microVMs every second. For an AI agent platform serving bursty traffic, this means:

  • A cold start that the user perceives as instant (125ms is faster than most API round-trips)
  • Ability to scale from zero to hundreds of concurrent sandboxes in seconds
  • No need to maintain warm pools as large as you would with slower VMs

Memory Overhead Breakdown

The under-5-MiB overhead covers:

  • The Firecracker process itself (Rust binary, statically linked)
  • Data structures for the 5 virtio device backends
  • KVM-related structures in the kernel
  • Page tables for the guest physical-to-host virtual memory mapping

This does not include the guest memory you allocate. If you give a VM 128 MiB for its workload, total consumption is 133 MiB. Compare that to a QEMU VM where the same workload consumes 478 MiB (350 MiB overhead + 128 MiB workload).

How Lambda Leverages Firecracker

AWS Lambda processes over 15 trillion invocations per month. Every one of those invocations runs inside a Firecracker microVM. When you deploy an AI coding agent as a Lambda function, the code your agent runs — including any untrusted code it generates and executes — is isolated at the hardware level from every other function on the same physical server.

The Lambda architecture uses Firecracker in a specific way:

  • One microVM per function instance. If your function scales to 100 concurrent executions, there are 100 separate Firecracker VMs running on the fleet.
  • Warm pools. Lambda keeps recently-used VMs warm so repeat invocations skip the boot time entirely. The 125ms cold start matters only for the first invocation or after idle timeout.
  • Snapshot/restore. Firecracker supports snapshotting a running VM's memory and CPU state, then restoring it later. This accelerates cold starts further by resuming from a pre-initialized state rather than booting from scratch.
  • vsock for control plane. Lambda's runtime communicates with the host via virtio-vsock, avoiding the need to configure networking just for internal control messages.

For agent sandboxes specifically, this means your agent can exec() arbitrary code, install packages, write to the filesystem, and even crash the guest kernel — none of which affects the host or other tenants. The CPU hardware enforces the boundary.

Example: Compare Resource Overhead of Different Isolation Approaches

The following example calculates how many isolated workloads you can fit on a 64 GB host using each approach. Each workload needs 128 MiB of application memory.

#!/usr/bin/env python3
"""Compare resource overhead of different isolation approaches."""

HOST_MEMORY_GB = 64
HOST_MEMORY_MIB = HOST_MEMORY_GB * 1024  # 65536 MiB

# (name, overhead_mib, app_memory_mib, startup_ms)
APPROACHES = [
    ("Process (fork)",          0,   128, 1),
    ("Container (Docker)",      30,  128, 500),
    ("Traditional VM (QEMU)",   350, 128, 5000),
    ("Firecracker MicroVM",     5,   128, 125),
]

for name, overhead, app_mem, startup in APPROACHES:
    total = overhead + app_mem
    instances = int(HOST_MEMORY_MIB // total)
    print(f"{name:<25} {overhead:>4} MiB overhead | "
          f"{instances:>4} instances | {startup:>5} ms startup")

Output:

Process (fork)                0 MiB overhead |  512 instances |     1 ms startup
Container (Docker)           30 MiB overhead |  414 instances |   500 ms startup
Traditional VM (QEMU)       350 MiB overhead |  137 instances |  5000 ms startup
Firecracker MicroVM           5 MiB overhead |  492 instances |   125 ms startup

Firecracker achieves 96% of the density of raw processes while providing hardware-level isolation. A traditional VM gives you only 27% of that density.
