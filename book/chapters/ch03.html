Chapter 3: Lambda MicroVMs Architecture

From Dockerfile to instant-launch sandbox with full lifecycle control

In Chapter 2 we explored how Firecracker provides hardware-level isolation in microseconds. Lambda MicroVMs build on that foundation, wrapping Firecracker in a managed service that handles image building, snapshotting, suspend/resume, and authentication for you. Announced on June 22, 2026 (launch blog post), the service is available in us-east-1, us-east-2, us-west-2, eu-west-1, and ap-northeast-1.

This chapter walks through the architecture end to end: how your Dockerfile becomes a bootable snapshot, how that snapshot powers instant launches, how the suspend/resume lifecycle keeps costs low while preserving state, and how authentication works. We finish with a comparison to traditional Lambda Functions and a hands-on lifecycle trace.

1. Image-Based Deployment: Dockerfile to Snapshot to Instant Launch

The deployment model for Lambda MicroVMs starts with two inputs you provide:

  1. A Dockerfile describing your sandbox environment (OS packages, runtimes, tools).
  2. A code artifact stored in S3 (your application code, agent harness, or bootstrap scripts).

When you call CreateMicrovmImage (API Reference), the service:

  1. Pulls the base image. AWS provides official base images: the ARN-based arn:aws:lambda:us-east-1:aws:microvm-image:al2023-1 or the public ECR image public.ecr.aws/lambda/microvms:al2023-minimal.
  2. Builds your Dockerfile on top of the base, incorporating the S3 code artifact.
  3. Boots the resulting image, runs initialization, then takes a memory snapshot of the fully-booted VM.
  4. Stores the snapshot for future launches.

Build logs stream to CloudWatch at /aws/lambda-microvms/<image-name>, so you can debug failed builds the same way you debug any CloudWatch log group.

The snapshot is the key insight. Because the VM is already booted and initialized in the snapshot, launching a new instance from it skips the entire boot sequence. The VM resumes from the snapshot in the same way your laptop resumes from sleep: memory contents, disk state, and process trees are all intact from the moment the snapshot was taken.

Resource Limits

Each MicroVM supports (Developer Guide):

  • Architecture: ARM64
  • vCPUs: up to 16
  • Memory: up to 32 GB
  • Disk: up to 32 GB
  • Max runtime: 8 hours

2. State Persistence Across Sessions

Unlike traditional Lambda Functions that start fresh on every invocation, a Lambda MicroVM preserves its full state between interactions:

  • Memory: Variables, heap allocations, in-memory caches all survive.
  • Disk: Files written to the filesystem persist. Databases, logs, downloaded dependencies stay put.
  • Processes: Long-running daemons, language servers, or background workers continue executing.

This makes MicroVMs suitable for agent sandboxes where an AI agent needs to install packages, build projects, run tests, and iterate - all within the same persistent environment across multiple interactions with a user.

When a MicroVM is suspended (see next section), this entire state is frozen and preserved. When it resumes, everything picks up exactly where it left off. The agent does not need to rebuild context, reinstall dependencies, or restore working directories.

3. The Suspend/Resume Lifecycle

A Lambda MicroVM moves through the following states:

CREATING ──▶ RUNNING ──▶ IDLE ──▶ SUSPENDED ──▶ TERMINATED
                ▲                      │
                └──────── RESUMED ◀────┘

Here is what drives each transition:

TransitionTriggerAPI Operation
CREATING → RUNNINGImage snapshot ready, VM bootedRunMicrovm
RUNNING → IDLENo incoming requests for a periodAutomatic (idle detection)
IDLE → SUSPENDEDIdle duration exceeds maxIdleDurationSecondsAutomatic, or SuspendMicrovm
SUSPENDED → RUNNINGNew request arrives (if autoResumeEnabled)Automatic, or ResumeMicrovm
Any → TERMINATEDExplicit termination or max runtime reachedTerminateMicrovm

Idle Policy Configuration

You control suspend behavior with three parameters:

  • maxIdleDurationSeconds - How long without requests before the VM suspends.
  • suspendedDurationSeconds - How long a suspended VM is kept before termination.
  • autoResumeEnabled - Whether incoming requests automatically wake a suspended VM.

Suspended instances incur no compute charges (product page). You pay only for the storage of the suspended state. This makes the suspend/resume model economical for agent sandboxes that see bursty, interactive traffic.

4. Authentication: JWE Tokens and Unique Endpoint URLs

When you call RunMicrovm, the service returns two things:

  1. A unique endpoint URL for that specific MicroVM instance.
  2. A JWE (JSON Web Encryption) authentication token.

All subsequent requests to the MicroVM must include the token in the X-aws-proxy-auth header. Tokens are short-lived; you obtain fresh ones via CreateMicrovmAuthToken.

# Obtain a fresh auth token for an existing MicroVM
aws lambda create-microvm-auth-token \
  --microvm-id mvm-0a1b2c3d4e5f67890

# Use the token to make requests to the MicroVM endpoint
curl -H "X-aws-proxy-auth: $TOKEN" \
  https://mvm-0a1b2c3d4e5f67890.microvm.lambda.us-east-1.amazonaws.com/exec

This auth model means each MicroVM is its own isolated endpoint. There is no shared gateway or multiplexed connection. An agent framework obtains the endpoint URL and token at launch time, then communicates directly with its sandbox. If the token expires mid-session, the framework calls CreateMicrovmAuthToken to get a new one without restarting the VM.

5. How It Differs from Lambda Functions

DimensionLambda FunctionsLambda MicroVMs
Execution modelEvent-driven, request/responseLong-running, interactive
StateStateless (fresh per invocation)Stateful (memory, disk, processes persist)
Max duration15 minutes8 hours
ConnectivityInvoked via trigger/SDKDirect HTTP endpoint per instance
Scaling modelConcurrent instances per requestOne instance per agent session
Cold startMilliseconds to secondsInstant (snapshot resume)
IsolationFirecracker MicroVM (shared)Dedicated Firecracker MicroVM per instance
Cost when idleNo charge (no instance exists)No compute charge (suspended state stored)
Use caseAPI backends, event processingAgent sandboxes, dev environments, long tasks

The key mental model shift: Lambda Functions are compute units that react to events, while Lambda MicroVMs are persistent machines you control directly. Functions scale horizontally and automatically; MicroVMs give you a dedicated, stateful environment with a direct network endpoint.

6. Example: Trace a MicroVM Lifecycle with AWS CLI

The following sequence shows the complete lifecycle using AWS CLI commands. Each step includes the command you would run and a summary of the response. (See examples/ch03_lifecycle_trace.py for a runnable script that prints this trace.)

# Step 1: Create an image from a Dockerfile + S3 code artifact
aws lambda create-microvm-image \
  --image-name agent-sandbox-v1 \
  --dockerfile-uri s3://my-bucket/sandbox/Dockerfile \
  --code-uri s3://my-bucket/sandbox/agent-code.tar.gz \
  --base-image arn:aws:lambda:us-east-1:aws:microvm-image:al2023-1
# Response: ImageArn, Status: BUILDING

# Step 2: Run a MicroVM from the image
aws lambda run-microvm \
  --image-arn arn:aws:lambda:us-east-1:123456789012:microvm-image:agent-sandbox-v1 \
  --vcpus 4 --memory-mb 8192 --disk-gb 16 \
  --idle-policy '{"maxIdleDurationSeconds":300,"autoResumeEnabled":true}'
# Response: MicrovmId, EndpointUrl, AuthToken, Status: RUNNING

# Step 3: Use the MicroVM (direct HTTP)
curl -X POST \
  -H "X-aws-proxy-auth: eyJhbGciOi..." \
  -H "Content-Type: application/json" \
  -d '{"command": "pip install pandas && python analysis.py"}' \
  https://mvm-abc123.microvm.lambda.us-east-1.amazonaws.com/exec

# Step 4: VM goes idle after 300s of no requests, then suspends automatically

# Step 5: Resume (automatic on next request, or explicit)
aws lambda resume-microvm --microvm-id mvm-abc123
# Response: Status: RUNNING (state fully restored)

# Step 6: Terminate when done
aws lambda terminate-microvm --microvm-id mvm-abc123
# Response: Status: TERMINATED

7. Challenge: Design a Lifecycle State Machine

Your task: implement a MicrovmLifecycle class that models the state machine described in this chapter. The class should:

  • Track the current state (CREATING, RUNNING, IDLE, SUSPENDED, TERMINATED).
  • Expose transition methods: run(), idle(), suspend(), resume(), terminate().
  • Raise InvalidTransitionError when a transition is not allowed from the current state.
  • Provide valid_transitions() returning the list of actions available from the current state.

The skeleton is in challenges/ch03.py. Tests are in challenges/ch03_test.py. A reference solution is in solutions/ch03.py.

Think carefully about which transitions are legal. For example: can you resume() from RUNNING? Can you terminate() from any state? Refer to the lifecycle diagram above.
