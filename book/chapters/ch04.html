Chapter 4

Your First MicroVM Image

From Dockerfile to bootable snapshot in under three minutes.

Prerequisites

Before you can build images, you need three things configured in your AWS account:

AWS CLI v2 with the lambda-microvms subcommand available. Verify with aws lambda-microvms help. If the subcommand is not present, update your CLI to the latest version.

An S3 bucket for storing your code artifacts. The build service pulls your application code from S3 during the image build process. Any bucket in the same region works.

A build role that the Lambda MicroVMs service can assume during the build. This role needs two permissions: read access to your S3 artifact, and write access to CloudWatch Logs for build output. The trust policy must allow lambda.amazonaws.com to assume the role, scoped to your specific image ARN.

{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": {"Service": "lambda.amazonaws.com"},
    "Action": "sts:AssumeRole",
    "Condition": {
      "StringEquals": {"aws:SourceAccount": "<your-account-id>"},
      "ArnEquals": {
        "aws:SourceArn": [
          "arn:aws:lambda:us-east-1:aws:microvm-image:al2023-1",
          "arn:aws:lambda:us-east-1:<your-account-id>:microvm-image:<your-image-name>"
        ]
      }
    }
  }]
}
⚠ Gotcha

The trust policy's ArnEquals condition must include both the base image ARN and your new image's ARN. If you only include the base image, the service can start the build but fails when it tries to write the resulting snapshot. The error message says "We were unable to assume the role provided" without explaining which step failed.

🔬 Under the hood

The build process does not just produce a Docker container. It runs your Dockerfile, starts the resulting container, waits for your CMD to initialize, then takes a Firecracker memory+disk snapshot of the running VM. When a user later calls RunMicrovm, they get a restored copy of that exact snapshot state. Your Flask app is already listening before the first request arrives.

The build is asynchronous. The service pulls your zip from S3, runs the Dockerfile, starts the resulting application, and snapshots the running state. Poll with get-microvm-image until the state transitions to CREATED (or UPDATED for subsequent versions).

In testing, this build completed in approximately 3 minutes for a minimal Flask application. The service builds two snapshots in parallel, one for each supported Graviton generation (3 and 4), so your image works on either chip without rebuilding.

🏭 In production

The list-microvm-image-builds command requires --image-version as a required parameter, not just the image identifier. If you are scripting around builds, you must first capture the version from the create/update response, then pass it when checking build status. This is a small but real gotcha for anyone automating the build pipeline.

The logs show the Docker build steps, package installation output, and any errors encountered during the build. If your build fails, this is where you find out why.

Common Build Failures

Two failure modes came up during testing of this chapter that are worth calling out:

"We were unable to assume the role provided." This means the build role's trust policy does not include your image ARN in its ArnEquals condition. The trust policy must list both the base image ARN and your new image's ARN.

"Access denied when fetching artifact from S3." The build role's IAM policy must grant s3:GetObject on the exact S3 object path of your artifact. A policy scoped to a different key path will fail silently (the build starts but cannot pull your code).

Both of these are permissions issues, not code issues. If your Dockerfile is correct but the build fails immediately, check IAM before anything else.

Your challenge is to implement two functions that validate and generate the inputs for a CreateMicrovmImage call. This is the kind of pre-flight validation your deployment tooling should do before sending a request to AWS.

Open challenges/ch04.py and implement:
