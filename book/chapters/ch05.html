Launching and Connecting

From image to running sandbox in five seconds, with authenticated access.

Launching with RunMicrovm

With an image built (Chapter 4), you can launch MicroVM instances from it. Each RunMicrovm call creates a dedicated VM restored from the image snapshot:

The response arrives immediately with state: PENDING. Within 5 seconds, GetMicrovm reports state: RUNNING. That 5-second window covers network routing setup on top of Firecracker's sub-125ms snapshot restore.

Note that maximumDurationInSeconds defaults to 28800 (8 hours). This is the absolute maximum lifetime of a single session, regardless of activity. After 8 hours the VM is terminated unconditionally.

⚠ Gotcha

The --idle-policy parameter requires all three fields together. If you omit suspendedDurationSeconds or autoResumeEnabled, the API rejects the request:

An error occurred (ParamValidation): Parameter validation failed:
Missing required parameter in idlePolicy: "suspendedDurationSeconds"
Missing required parameter in idlePolicy: "autoResumeEnabled"

You might reasonably assume these are independently optional. They are not. All three or none.

Resource Configuration

Each MicroVM instance can be configured with up to 16 vCPUs, 32 GB memory, and 32 GB disk. These limits are set at the image level (during CreateMicrovmImage) rather than at launch time.

The default configuration allocates 2048 MiB of memory (the minimum). For compute-heavy workloads like compilation or data processing, configure higher resources during the image build:

aws lambda-microvms create-microvm-image \
  --name heavy-compute-image \
  --base-image-arn arn:aws:lambda:us-east-1:aws:microvm-image:al2023-1 \
  --build-role-arn arn:aws:iam::<your-account-id>:role/my-build-role \
  --code-artifact uri=s3://bucket/artifact.zip \
  --resources '[{"minimumMemoryInMiB": 8192}]'

The default network connectors provide HTTP ingress and internet egress without any explicit configuration. You get a unique HTTPS endpoint per MicroVM instance and full outbound internet access.

Authentication with JWE Tokens

MicroVMs are not publicly accessible. Every request to the endpoint must carry a short-lived token in the X-aws-proxy-auth header. You create tokens with CreateMicrovmAuthToken:

The token is a JWE (JSON Web Encryption) compact serialization string with five dot-separated base64url segments: protected header, encrypted key, IV, ciphertext, and authentication tag. This is a real encryption envelope, not just a signed JWT.

⚠ Gotcha

The --allowed-ports parameter is a tagged union. Each entry must contain exactly one key: port, range, or allPorts. Adding a protocol field (which you might expect to exist) produces this error:

An error occurred (ParamValidation): Parameter validation failed:
Invalid number of parameters set for tagged union structure allowedPorts[0].
Can only set one of the following keys: port. range. allPorts.
Unknown parameter in allowedPorts[0]: "protocol", must be one of: port, range, allPorts

The valid shapes are: {"port": 5000}, {"range": {"start": 8000, "end": 9000}}, or {"allPorts": true}.

🔬 Under the hood

The *.lambda-microvm.us-east-1.on.aws endpoint is a per-instance HTTPS hostname. AWS handles TLS termination and routes the request to your specific MicroVM after validating the JWE token. The token is decrypted at the routing layer, never by your application code. Your app just sees a normal HTTP request on the port you specified.

Inspecting MicroVM Status

Use GetMicrovm at any time to check the current state, resource allocation, and timing information:

aws lambda-microvms get-microvm --microvm-identifier microvm-29fabacb-68fe-30ed-b477-39bf36e55b16

The response includes the current state (PENDING, RUNNING, SUSPENDED, or TERMINATED), the endpoint URL, idle policy, network connectors, and the startedAt timestamp. This is your primary observability tool for individual MicroVM instances.

For listing all instances across your account, use ListMicrovms. Terminated instances remain in the list with minimal metadata (no endpoint or policy fields), which is useful for auditing session history.

Challenge: Build RunMicrovm and Auth Token Helpers

Your challenge is to implement three helper functions that construct API parameters and parse responses. These are the kind of SDK wrapper functions you would write to avoid the gotchas we encountered above.

Open challenges/ch05.py and implement:

  • build_run_microvm_params() - constructs RunMicrovm parameters, always including all three idle policy fields
  • build_auth_token_params() - constructs auth token parameters, validating the tagged-union port spec
  • parse_microvm_status() - parses a GetMicrovm response into a simplified status summary

cd challenges && python3 -m pytest ch05_test.py -v
