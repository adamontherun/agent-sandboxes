Chapter 6

Lifecycle Management

Suspend idle sessions to stop the meter, resume them instantly when needed, and terminate gracefully when done.

Manual Suspend and Resume

A running MicroVM costs money every second it is active. When a user walks away for lunch, you do not want to keep paying for an idle VM. The suspend/resume mechanism lets you pause a session without losing any state:

Both SuspendMicrovm and ResumeMicrovm return no response body on success. This is confirmed behavior from a live test, not a documentation gap. The state transition completes within about 3 seconds, verified via a follow-up GetMicrovm call.

🔬 Under the hood

Suspension takes a Firecracker snapshot of the VM's memory and disk state, then releases the underlying compute. Resume restores from that snapshot. Because Firecracker's snapshot restore runs in under 125ms, the resume is nearly instantaneous from the user's perspective. The 3-second wall-clock time includes network re-routing overhead on top of the raw restore.

What Survives Suspension

When a MicroVM is suspended and later resumed, the following state is preserved:

  • Memory - all process memory, including heap allocations, global variables, and stack frames
  • Disk - the full filesystem, including any files written during the session
  • Processes - all running processes resume from exactly where they paused, including open file descriptors and network sockets

From the application's perspective, time appears to skip forward. A time.time() call before suspend and after resume will show the elapsed wall-clock time, but no code executed during that gap. Network connections established before suspension may time out on the remote end during the suspended period, so applications should handle reconnection gracefully.

🏭 In production

If your application maintains long-lived connections (WebSockets, database pools, gRPC streams), add reconnection logic that triggers on resume. A simple pattern: catch connection errors on first use after any idle period and re-establish automatically.

Auto-Suspension and Cost Optimization

The idle policy you set at launch time controls automatic suspension. The three fields work together as a state machine:

  1. maxIdleDurationSeconds - after this many seconds with no incoming requests, the MicroVM is automatically suspended
  2. suspendedDurationSeconds - after this many seconds in the SUSPENDED state (with no resume), the MicroVM is terminated
  3. autoResumeEnabled - if true, an incoming request to a SUSPENDED MicroVM triggers automatic resume before the request is forwarded

A typical configuration for an interactive coding assistant:

{
    "maxIdleDurationSeconds": 300,
    "suspendedDurationSeconds": 3600,
    "autoResumeEnabled": true
}

This means: suspend after 5 minutes idle, terminate after 1 hour suspended, and auto-resume if the user comes back. The user experiences a brief delay (a few seconds) on the first request after idle, then full speed from there.

For batch workloads that should not auto-resume (you want them to finish their current work and stop):

{
    "maxIdleDurationSeconds": 60,
    "suspendedDurationSeconds": 60,
    "autoResumeEnabled": false
}

This aggressively reclaims resources: 1 minute idle triggers suspension, 1 minute suspended triggers termination, and no automatic wake-up.

Graceful Termination

Termination is permanent. Once terminated, a MicroVM cannot be resumed. All state is destroyed.

After termination, the instance remains in ListMicrovms with state: TERMINATED but only retains minimal metadata. The endpoint URL, idle policy, and network connector fields are absent from terminated entries:

This is useful for audit trails and billing reconciliation. You can see when each session started and which image it ran, even after the session is gone.

Handling Failures and Timeouts

Several failure modes to handle in your orchestration layer:

Suspend of a non-running instance. Calling SuspendMicrovm on a PENDING or already-SUSPENDED instance will fail. Always check state with GetMicrovm before issuing lifecycle commands, or handle the error gracefully.

Resume timeout. If a resume takes longer than expected (rare, but possible under high load), poll GetMicrovm rather than assuming instant state transition. A 10-second timeout with 3-second polling intervals is a reasonable default.

The 8-hour wall clock. The maximumDurationInSeconds (default 28800) is an absolute ceiling. Even if the MicroVM has been suspended for most of its life, the clock ticks from startedAt to termination. Plan for session rotation if your use case needs longer-lived environments.

Terminate during operations. You can terminate a MicroVM in any non-terminal state (PENDING, RUNNING, SUSPENDED). This is your escape hatch if something goes wrong. Always prefer explicit termination over relying solely on idle-policy timeouts for cleanup.

🏭 In production

Build your orchestrator with a "terminate on cleanup" guarantee. Use try/finally patterns (or equivalent in your language) to ensure every launched MicroVM gets terminated, even if your control plane crashes. A stale running MicroVM is a billing leak. Run periodic ListMicrovms sweeps to catch any that escaped your cleanup logic.

Challenge: Implement a Lifecycle Policy Engine

Your challenge is to implement the decision logic that an orchestrator uses to manage MicroVM state transitions. Given the current state of a MicroVM (idle duration, suspended duration, whether a request is waiting) and its configured policy, determine what action to take.

Open challenges/ch06.py and implement:

  • should_suspend() - decide if a running MicroVM should be suspended based on idle time
  • should_terminate() - decide if a suspended MicroVM should be terminated
  • should_auto_resume() - decide if a suspended MicroVM should be woken up
  • compute_lifecycle_action() - combine the above into a single decision function

Run the tests:

cd challenges && python3 -m pytest ch06_test.py -v

The test suite includes 21 cases covering each decision function independently and the combined logic, including edge cases like resume-takes-priority-over-terminate (if a user returns just as the terminate timer fires, the user wins). See solutions/ch06.py when you are ready to compare.
