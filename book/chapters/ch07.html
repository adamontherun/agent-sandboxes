Chapter 7

Executing Untrusted Code

Receive a snippet, run it, get the output back, and understand exactly which layer is stopping it from doing something worse.

The shape of a code execution endpoint

Everything in Part II got you a running MicroVM with a live endpoint. This chapter is about what that endpoint actually does when it is the backing service for an AI agent: accept a code snippet over HTTP, run it, and hand back what happened. Strip away the AWS-specific plumbing and the pattern is the same one every "run this for me" sandbox uses:

  1. Receive the snippet in a request body.
  2. Write it to a file.
  3. Execute that file in a subprocess, with a timeout.
  4. Capture stdout, stderr, and the exit code.
  5. Clean up the file, regardless of how execution went.

None of these five steps is specific to Lambda MicroVMs. You could run this exact Flask app on a laptop, in a Docker container, or inside a MicroVM, and the code would not change. What changes is what surrounds it, and that is the part worth being precise about: it is easy to write an endpoint like this and quietly assume it is "sandboxed" when the sandboxing is actually coming from somewhere else entirely (or from nowhere).

⚠ Gotcha

This chapter's example script runs on a local dev machine, not inside a real MicroVM: there is no AWS account interaction here, unlike Chapters 4–6. The subprocess-level isolation shown below is real and testable locally, but it is not the security boundary. Read the "What the MicroVM actually isolates" section before assuming this endpoint alone is safe to expose to untrusted users.

Designing the endpoint

A minimal execution endpoint takes a JSON body with the code to run and an optional timeout, and returns what came out the other side:

@app.post("/execute")
def execute_endpoint():
    """POST {"code": "...", "timeout_seconds": 5} -> execution result."""
    payload = request.get_json(force=True, silent=True) or {}
    code = payload.get("code", "")
    timeout_seconds = float(payload.get("timeout_seconds", 5.0))

    if not code:
        return jsonify({"error": "missing 'code' field"}), 400

    result = execute_snippet(code, timeout_seconds=timeout_seconds)
    return jsonify({
        "stdout": result.stdout,
        "stderr": result.stderr,
        "exit_code": result.exit_code,
        "timed_out": result.timed_out,
        "duration_seconds": result.duration_seconds,
    })

The interesting work is in execute_snippet, not the route handler. The route handler's job is small on purpose: parse the request, reject an obviously bad one, delegate, and serialize the result. Keeping the HTTP layer thin makes the execution logic testable without spinning up Flask at all, which is exactly what this chapter's challenge does.

Writing to disk, executing, capturing output

The full execute_snippet function writes the incoming code to a temp file, runs it with subprocess.run, and captures everything that comes back:

def execute_snippet(code: str, timeout_seconds: float = 5.0) -> ExecutionResult:
    workdir = Path(tempfile.mkdtemp(prefix="ch07-exec-"))
    script_path = workdir / f"{uuid.uuid4().hex}.py"
    script_path.write_text(code)

    started = time.monotonic()
    timed_out = False
    try:
        proc = subprocess.run(
            [sys.executable, str(script_path)],
            capture_output=True,
            text=True,
            timeout=timeout_seconds,
            cwd=workdir,
        )
        stdout, stderr, exit_code = proc.stdout, proc.stderr, proc.returncode
    except subprocess.TimeoutExpired as exc:
        timed_out = True
        stdout = exc.stdout or ""
        stderr = (exc.stderr or "") + "\n[execution timed out]"
        exit_code = None
    finally:
        duration = time.monotonic() - started
        script_path.unlink(missing_ok=True)
        workdir.rmdir()

    return ExecutionResult(stdout=stdout, stderr=stderr, exit_code=exit_code,
                            timed_out=timed_out, duration_seconds=round(duration, 3))

A few details worth calling out explicitly:

  • Each snippet gets its own temp directory and a UUID-named file, not a shared path. Two concurrent requests never collide, and there is no snippet left over from a previous request for the next one to accidentally read.
  • The finally block deletes the script and its directory whether the process succeeded, crashed, or timed out. Cleanup that only runs on the happy path is not cleanup.
  • cwd=workdir means a snippet that does something like open("output.txt", "w") writes into the disposable temp directory, not wherever the Flask process happens to be running from.

The endpoint's job is to report what the snippet did, including "it threw an exception," not to prevent every possible failure inside the snippet.

Timeout handling and resource limits

The most important line in execute_snippet is easy to skim past: timeout=timeout_seconds on the subprocess.run call. Python's own documentation is precise about what this does: if the timeout expires, the child process is killed and waited for, and TimeoutExpired is re-raised after the child has actually terminated. That guarantee, that the process is dead before your code continues, is what makes an infinite loop a non-event instead of a leaked runaway process:

A one-second timeout was passed for this snippet specifically (the rest use the five-second default); the measured duration of 1.003s confirms the kill happened right at the boundary, not some arbitrary amount of time later. exit_code comes back None because the process was killed rather than exiting on its own: that is the caller's signal to distinguish "the code returned a nonzero status" from "the code never returned at all."

A wall-clock timeout is necessary but not sufficient. subprocess.run's timeout stops a snippet from running forever; it does nothing about a snippet that allocates memory until the host falls over, forks child processes it never cleans up, or opens far more file descriptors than it needs. Those require resource limits, not just a clock, which is exactly the gap a MicroVM (or, on a self-managed Firecracker host, the jailer) closes rather than the application code.

🔬 Under the hood

On a self-hosted Firecracker deployment, per-instance resource limits are enforced through cgroups: memory.limit_in_bytes for memory, cpu.cfs_quota_us for CPU share, and blkio.throttle.io_serviced for disk I/O, plus the jailer's own fsize limit on file size, applied via setrlimit(). Lambda MicroVMs manage this for you: the memory, vCPU, and disk figures you set at image build time are the resource ceiling the platform enforces, not a suggestion your application code has to police itself.

Security boundaries: what the MicroVM isolates

This is the question the chapter's opening gotcha promised to answer: if subprocess.run with a timeout is not the security boundary, what is?

Firecracker's own project page is explicit that a MicroVM is not just a process-level sandbox: "Firecracker microVMs use KVM-based virtualizations that provide enhanced security over traditional VMs." KVM-based virtualization means the untrusted code is running behind actual hardware virtualization support, the same category of isolation that separates one cloud customer's EC2 instance from another's on the same physical host. It is not merely a different Linux user or a different container namespace; it is a different virtual machine with its own kernel.

Firecracker adds a second layer on top of that hardware boundary. The project describes a companion program, the jailer, that wraps each microVM with conventional Linux sandboxing on top of the virtualization: "Each Firecracker microVM is further isolated with common Linux user-space security barriers by a companion program called 'jailer'," which "provides a second line of defense in case the virtualization barrier is ever compromised." Concretely, per the jailer's own design (see Chapter 2's research notes), that means a chroot via pivot_root(), dedicated mount/PID/network namespaces, and a non-privileged uid/gid the Firecracker process runs as after setup: defense in depth, not a single wall.

Put together, here is what actually stops a malicious snippet inside a Lambda MicroVM, and what does not:

  • The MicroVM isolates: the kernel (a MicroVM has its own, not the host's), memory and CPU (enforced by the platform's resource configuration, not by application code), the filesystem (the MicroVM's disk is its own snapshot, not a bind-mount into the host), and process/network namespaces from the host and from other tenants' MicroVMs.
  • The MicroVM does not automatically isolate: the snippet from other requests running inside the same MicroVM at the same time, or a snippet from a badly-scoped IAM role attached to that MicroVM's execution role. A shared MicroVM serving many users' snippets sequentially still needs application-level cleanup between requests (Chapter 9 covers this); a MicroVM's network egress still reaches wherever its network connectors point unless you've deliberately restricted them (Chapter 8 touched on this for package registries; Chapter 12 covers hardening it further).

This chapter's execute_snippet function, running on a local dev machine, gives you exactly the "capture output and enforce a wall-clock timeout" part of that list, and nothing else. There is no separate kernel here, no memory ceiling, no filesystem boundary beyond ordinary OS file permissions. That is an honest limitation of running this example locally to write and test it quickly; deployed inside a real Lambda MicroVM, the same Flask code gets the rest of the list for free, supplied by the platform underneath it rather than by anything this chapter's Python had to implement.

🏭 In production

Do not write your own process-isolation logic and call it a sandbox. The value of a MicroVM-backed executor is precisely that the isolation guarantee lives below your application code, where a bug in your Python can't undermine it. If you find yourself adding seccomp filters or namespace tricks inside the code-execution endpoint itself, that is usually a sign the endpoint is running somewhere that isn't actually providing VM-level isolation.

Challenge: Build a Python code executor

Your challenge is to implement the decision logic behind a code-execution endpoint, factored so it can be tested without spawning real subprocesses. Open challenges/ch07.py and implement:

  • validate_request(payload): check an incoming request body for a non-empty code string and an optional positive timeout_seconds, returning an error message or None.
  • build_command(script_path): build the subprocess argv used to run a saved script.
  • summarize_result(stdout, stderr, exit_code, timed_out): normalize raw subprocess output into an ExecutionResult.
  • is_successful(result): decide whether an execution counts as successful (exit code 0 and no timeout; a timeout with a coincidental zero exit code still does not count).

Run the tests:

cd challenges && python3 -m pytest ch07_test.py -v

The suite has 18 cases across request validation, command construction, result normalization, and the success predicate. See solutions/ch07.py when you are ready to compare.
