Chapter 8

Package Management and Dependencies

Bake it now or install it later — and why the answer changes what your sandbox can do.

Two places dependencies can live

Every package your agent uses has to arrive somewhere. In a MicroVM sandbox, "somewhere" is one of two places: burned into the image at build time, or fetched by pip install from inside a running MicroVM. Both work. They fail differently, cost differently, and constrain your agent differently.

This chapter is about picking the right place for each dependency. The mechanics are simple. The judgment is where the money is.

Bake vs. install

Baking means the package is installed in the Dockerfile that produces your MicroVM image. It is present in the snapshot. Every MicroVM launched from that image starts with the package already imported and ready. The tradeoff: rebuilding the image takes minutes, and every extra megabyte lives in the snapshot forever.

Installing at runtime means the MicroVM starts clean and your agent (or your bootstrap script) runs pip install some-package after the VM is live. The tradeoff: you pay the install cost on every fresh MicroVM, and you need network egress to reach PyPI.

Persistence changes the math

Here is what makes Lambda MicroVMs different from a normal container: state persists across suspend and resume. When you call SuspendMicrovm, the running memory and disk are snapshotted. When you later call ResumeMicrovm, that exact state comes back. A package you installed at runtime is still there.

So the pattern is: install once, use many times. A long-lived sandbox that you keep suspending and resuming behaves more like a bakery than a fresh cluster — the packages you installed on Monday are still available on Friday.

⚠ Gotcha

Suspend and resume preserve the packages. Termination does not. If a MicroVM is terminated (out of TTL, out of memory, or by explicit TerminateMicrovm), everything you installed at runtime is gone. The next launch from the same image starts clean. Do not assume runtime installs are durable across the MicroVM's lifetime unless you understand who controls termination.

Caching strategies

If you install at runtime, cache the wheels. A pip cache lives at ~/.cache/pip and is preserved by suspend/resume just like the packages themselves. On a warm resume, pip install pandas pulls from the local cache in under a second instead of downloading from PyPI.

Three strategies, from cheapest cold start to most flexible:

  • Bake everything. No network dependency, no install latency. Best for stable dependency sets that change on a build cycle.
  • Bake the base, install the tail. Common packages (Flask, boto3, requests) in the image; rare or agent-specific packages installed at runtime. This is the pattern most production sandboxes converge on.
  • Install everything, cache aggressively. Fresh image on each launch, but a warm pip cache on suspend/resume. Best when the dependency set is genuinely unpredictable, e.g. an agent that installs whatever the user asks it to.
🏭 In production

Baking the ~20 most common packages into the image cuts perceived latency on fresh MicroVMs from tens of seconds to sub-second. The rule of thumb: any package you import on more than half of agent runs belongs in the image. Reserve runtime install for the long tail.

Network access for PyPI

Runtime pip install needs to reach pypi.org. Lambda MicroVMs are network-isolated by default; you attach network access through egress connectors on the image. The managed INTERNET_EGRESS network connector gives outbound HTTPS to arbitrary internet hosts.

If you bake all dependencies and never install at runtime, you can omit the internet egress connector entirely. That is a real security posture upgrade — an agent that cannot reach PyPI also cannot exfiltrate data to an attacker-controlled host. Keep egress off unless you need it.

Challenge: Plan the install list

Your challenge is to implement the logic that decides which packages a running MicroVM needs to install. Given a list of requested packages and the set already baked into the image, produce a plan: what is available, what needs installing, and the exact pip install command to run.

Open challenges/ch08.py and implement:

  • parse_requirements(requirements_text) — parse a requirements.txt-format string, handling comments, blanks, version specifiers, extras, and environment markers. Returns lowercased package names.
  • resolve_dependencies(requested, baked) — return a DependencyPlan splitting packages into already_available, needs_install, and an install_command string.

Run the tests:

cd challenges && python3 -m pytest ch08_test.py -v

The suite has 15 cases covering parsing edge cases and the resolver. Compare with solutions/ch08.py when you are ready.

⚡ Try it

The runnable example at examples/ch08_package_management.py feeds a realistic requirements.txt through the resolver and prints the install command that a bootstrap script would run inside a fresh MicroVM. No AWS calls; run it locally.
