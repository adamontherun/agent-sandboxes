Chapter 9

File System Isolation

The disk is real and it persists, which means user-supplied paths and leftover files are now your problem.

The MicroVM's filesystem is not a sandbox by default

A Lambda MicroVM gives you a real disk, up to 32 GB per the documented resource ceilings, backed by a virtio-block device. From inside the guest it looks and behaves like any other Linux filesystem: open(), write(), and os.listdir() work exactly as they do on your laptop. That is convenient and dangerous in equal measure.

It is convenient because your agent's code doesn't need to know it is running in a MicroVM to read and write files. It is dangerous because nothing about the filesystem itself stops a user-supplied path from walking out of the directory you meant to confine it to. The MicroVM boundary protects you from other tenants' MicroVMs; it does nothing to stop ../../etc/passwd from resolving to a real file inside your own guest.

The two problems this chapter solves are the same shape: something outside your control, a user-supplied path, or an accumulation of files you never explicitly cleaned up, interacts with your workspace directory. You handle both by treating the workspace as a hostile boundary and writing code that enforces its edges.

Persistent storage across requests

As covered in Chapter 8, Lambda MicroVMs preserve disk state across SuspendMicrovm and ResumeMicrovm. That means a workspace directory you write to during one request is still there, byte for byte, on the next request, as long as the MicroVM was suspended rather than terminated.

This is what makes file-based patterns useful for agents at all. An agent that writes intermediate results, uploaded files, or generated artifacts to /workspace can rely on those files surviving between turns of a conversation. Without persistence, every request would need to re-upload or regenerate everything from scratch.

The corollary: persistence means accumulation. A workspace directory that nobody cleans up grows forever across a long-lived session, one suspend/resume cycle at a time. That is the problem the rest of this chapter is about.

Safe path resolution

Any time your agent runtime accepts a path from an untrusted source, a filename in a tool call, a field in a request body, an argument the model generated, that path has to be resolved against your workspace root before you touch the filesystem with it. The naive approach, os.path.join(root, user_path), does not protect you: os.path.join("/workspace", "../etc/passwd") happily returns /workspace/../etc/passwd, and every filesystem call that follows will walk right out of the workspace.

The fix is to resolve the joined path with os.path.realpath, which collapses .. segments and follows symlinks, and then check that the result is still inside the root:

Three checks matter, in order. First, reject an absolute user_path outright: /etc/passwd should never even reach the join, because os.path.join discards everything before an absolute second argument and would silently return /etc/passwd untouched. Second, resolve with realpath so that subdir/../../etc/passwd collapses to its real, final destination instead of a string that merely looks contained. Third, compare the resolved target against the resolved root using a trailing separator (root_resolved + os.sep), not a bare prefix check. Without the separator, a root of /workspace would incorrectly accept a sibling directory like /workspace-evil.

Running the example against a temporary sandbox root shows the three traversal attempts rejected and the three legitimate paths (including one with a harmless internal .. that never leaves the root) resolved cleanly:

Note the third accepted case: subdir/../top.py resolves to top.py directly under the root. That is correct: the path never actually leaves the root, it just takes a detour through a directory that does not exist. safe_resolve only rejects paths whose final destination is outside the root, not paths that merely contain a .. segment.

⚠ Gotcha

realpath also resolves symlinks. If your workspace directory can contain a symlink planted by a previous (possibly malicious) request, say, a file named output.txt that is actually a symlink to /etc/passwd, a naive traversal check on the string alone won't catch it, but realpath will, because it follows the symlink before comparison. This is one more reason to resolve with realpath rather than string manipulation: it collapses both .. segments and symlink indirection to the same real destination.

Cleaning up user-generated files

Every artifact your agent writes, scratch files, logs, cached build output, is a candidate for cleanup between executions. But not everything should go: a workspace often has files worth preserving across a cleanup pass, such as a persistent scratch script or a checked-out repository the agent is actively working in.

Two details keep this correct. Symlinks are treated as files, via os.unlink, not a recursive walk, so a symlink into another directory never causes cleanup to wander outside the entry itself. And directories are sized before they are deleted: os.walk has to visit every file underneath to add up bytes_freed, then shutil.rmtree removes the whole subtree in one call. Getting the order backwards would report zero bytes freed for every directory.

14,336 bytes freed is the sum of output.log (2,048 bytes), cache.bin (4,096 bytes), and the build.tar file nested inside artifacts/ (8,192 bytes). The directory's own metadata contributes nothing; only the files underneath it do.

Quota management and disk limits

Lambda MicroVMs cap total disk at 32 GB, but that ceiling is set once at image-build time and is not something your application code can query or adjust at runtime. Anything finer, per-workspace quotas, per-user limits, warning thresholds, is a layer you build yourself on top of that hard ceiling, the same way Chapter 10's resource limits layer OS-level guards inside a coarser MicroVM boundary.

The building blocks are the two functions above plus one more measurement: walk the workspace, sum file sizes, and compare against your own configured limit before accepting new writes.

Check the quota before the write, not after. A write that fails partway through because the underlying disk filled up leaves a truncated file behind, and now your next quota check has to account for a corrupt partial artifact rather than a clean rejection.

🏭 In production

Run cleanup_workspace proactively, not just reactively. A common pattern is to call it at the start of every new task with a preserve set scoped to files the current task actually needs, rather than waiting for a quota check to fail. That converts an occasional hard rejection into routine housekeeping the user never notices.

Reading uploaded files and artifacts

The read path deserves the same discipline as the write path. When an agent tool accepts a filename to open, "read the file the user uploaded as report.csv", that filename is exactly as untrusted as a filename to write to. Route it through safe_resolve first:

Two failure modes matter beyond traversal. A path that resolves inside the root but points at a directory, not a file, should fail with a clear error rather than an IsADirectoryError bubbling up from open(). And a file that is technically inside the workspace but far larger than your agent runtime should ever load into memory, a multi-gigabyte log an earlier task left behind, needs its own size check, independent of the aggregate quota, before you read it whole.

Challenge: Build a file-based code runner

Your challenge is to implement the two functions this chapter is built around. Open challenges/ch09.py and implement:

  • safe_resolve(root, user_path): resolve a user-supplied relative path against an absolute root, rejecting empty paths, absolute paths, relative roots, and any traversal that would resolve outside the root. Returns the resolved absolute path.
  • cleanup_workspace(workspace_dir, preserve=None): remove every top-level file and directory in workspace_dir except names in preserve, returning a CleanupResult with removed, preserved, and bytes_freed.

cd challenges && python3 -m pytest ch09_test.py -v

The suite has 15 cases: eight covering safe_resolve's traversal, absolute-path, empty-path, and relative-root rejections, and seven covering cleanup_workspace's removal, preservation, directory recursion, byte accounting, and error handling. Compare with solutions/ch09.py when you are ready.
