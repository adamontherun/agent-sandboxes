Chapter 10

Resource Limits and Quotas

Size MicroVMs at launch, layer OS-level guards inside them, and treat the 8-hour wall clock as an outer bound rather than your budget.

Sizing the MicroVM

Memory is set on the image via --resources, not the launch call, and stays fixed for the lifetime of any MicroVM launched from it. You cannot resize a running instance; if you need more, you build a differently-sized image and launch from that instead. You don't set a vCPU count directly — vCPU scales with memory at roughly 1 vCPU per 2 GB (an 8 GB image gets about 4 vCPUs). The documented ceilings are 32 GB of memory and, at that ratio, 16 vCPUs per MicroVM.

Pick the smallest size that fits your workload's steady state. Agents running Python and shelling out to gcc or node spend most of their time waiting on I/O, so 4–8 GB (roughly 2–4 vCPUs) is often plenty. Machine-learning inference or heavy compilation is where the upper end starts to matter. If your workload's needs change, that's a new image build, not a flag on an existing one.

All three idlePolicy fields shown above are required together when launching — Chapter 5 covers why omitting any one of them fails validation.

The 8-hour Wall Clock

Every MicroVM has a hard maximum lifetime. The parameter maximumDurationInSeconds accepts values up to 28800 (8 hours), and 28800 is also the default when unspecified. Once that ceiling is reached, the MicroVM is terminated by the platform regardless of what it is doing.

The clock starts at startedAt, returned when the MicroVM launches. The documentation doesn't spell out whether suspension pauses this clock, so treat it as running continuously until you've verified otherwise against your own account: plan as if a MicroVM suspended for 7 of its 8 hours could still be terminated at the 8-hour mark. For long-lived environments, plan a session-rotation strategy regardless: snapshot state to durable storage, launch a fresh MicroVM, and rehydrate.

⚠️ Gotcha

Do not treat maximumDurationInSeconds as your per-request timeout. It is the ceiling for the entire session. A single runaway task can happily consume the full 8 hours of billed compute if that is your only guard. Always add a shorter per-request timeout at the orchestrator level.

Defense in Depth Inside the Guest

The MicroVM boundary is enforced by Firecracker and is the primary line of defense. But it is coarse: it caps the whole VM. When a single MicroVM handles many small requests, or when you want to fail a specific runaway task without killing the session, you also need limits inside the guest.

Two complementary tools:

  • Shell ulimit for setting limits on child processes launched by the agent. Useful for capping CPU time, file size, and process count for a subshell that runs untrusted code.
  • Python resource.setrlimit for programmatic control from the runtime that supervises those child processes.

A classic fork bomb, :(){ :|:& };:, exhausts process table slots by forking recursively. Without RLIMIT_NPROC, it can starve the guest even though the VM boundary would eventually catch it via memory exhaustion. With RLIMIT_NPROC=32, the 33rd fork fails and the parent shell keeps running.

🔬 Under the hood

Firecracker uses Linux cgroups (via its jailer) to enforce hard memory and CPU limits at the VM boundary. Your Python setrlimit and shell ulimit calls operate on the process boundary inside that VM. Cgroups are the safety net; rlimits are the fast, precise guard for individual tasks. You want both.

Per-Request Timeouts

Every request into the sandbox needs its own timeout, independent of the MicroVM's max duration. A reasonable layering:

  1. Client timeout (seconds to a few minutes). Set on the HTTP client that calls into the MicroVM. Bounds how long an upstream caller waits.
  2. Server-side task timeout. Enforced inside the guest by the agent runtime, using signal.alarm, asyncio.wait_for, or a subprocess timeout=.
  3. MicroVM idle policy (Chapter 6). Reclaims the whole VM after a longer idle period.
  4. Max duration (8 hours). The absolute ceiling.

Each layer is redundant on purpose. If the agent hangs but the socket stays open, the client timeout fires. If the client hangs on to a broken connection, the task timeout fires. If both fail, the idle policy eventually reclaims the VM. If all three fail, the 8-hour ceiling catches it. You do not want to rely on any single layer.

Monitoring What You Configured

Configured limits are only useful if you can see when they bite. A minimal monitoring surface:

  • Log every PolicyDecision your orchestrator makes: which requests were denied, which were clamped, which sailed through.
  • Emit CloudWatch metrics keyed by rejection reason: PolicyDenied{reason="blocked_pattern"}, PolicyClamped{field="timeout"}. Deny-rate spikes almost always precede either abuse or a legitimate user hitting a limit that is now too tight.
  • On the guest side, capture exit codes and signals. A subprocess killed by SIGKILL with RLIMIT_CPU exceeded, or MemoryError from RLIMIT_AS, are distinct signals worth counting.
🏭 In production

Alert on the ratio of clamped-to-total requests, not the absolute count. A sudden climb from 2% clamped to 40% clamped usually means either policy drift (limits set too tight for real traffic) or a caller sending malformed requests. Both are worth a page.

Challenge: A Policy Evaluator for Incoming Requests

Your challenge is to implement the front-door check that every request goes through before an orchestrator hands it to a MicroVM. It is the piece that decides whether a request is safe, whether its resource asks are within bounds, and whether to clamp them silently or reject the request outright.

Open challenges/ch10.py and implement evaluate_request(request, policy). It receives an ExecutionRequest (code, timeout, memory) and a ResourcePolicy, and returns a PolicyDecision:

  • Deny (with a reason) if the code contains any pattern from policy.blocked_patterns.
  • Deny if the code's UTF-8 byte size exceeds policy.max_code_size_bytes.
  • Clamp timeout and memory to the policy's bounds; report clamps via adjusted_timeout / adjusted_memory. Clamping does not deny.
  • Otherwise, allow.

Run the tests:

cd challenges && python3 -m pytest ch10_test.py -v

Eleven tests cover the allow path, both deny paths, all four clamp directions, custom pattern lists, and the shape of the return value. See solutions/ch10.py when you are ready to compare.

⚡ Try it

Run examples/ch10_resource_limits.py to see the reference solution evaluate a range of requests — normal, fork-bomb, oversized, and out-of-bounds — with no AWS calls required.
