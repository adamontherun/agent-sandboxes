Chapter 11

Multi-Tenant Isolation

One tenant per MicroVM buys the strongest isolation guarantee AWS offers. A shared pool buys lower cost. Most real systems need both.

One MicroVM per User vs. Shared MicroVMs

Every chapter so far has assumed a single MicroVM serving a single workload. Once you have real tenants (different users, different customers, different API keys), you have to decide how many of them share a MicroVM, and the two ends of that spectrum trade off very differently.

Dedicated per tenant gives each tenant their own MicroVM, launched on first use and held for as long as that tenant is active. Firecracker's hardware-level isolation (Chapter 2) then applies at the tenant boundary, not just the request boundary: one tenant's process table, filesystem, and memory are never visible to another, because they are never in the same guest kernel. This is the isolation model AWS itself calls out: the product page names per-user and multi-tenant isolation among its intended use cases, and notes/research.md's summary of that page describes the underlying guarantee as "dedicated MicroVM per session, no shared kernel between users."

Shared pool instead routes many tenants through a small, fixed set of MicroVMs, reusing an instance across tenants over time. This costs less (you are not paying to keep hundreds of mostly-idle MicroVMs warm), but it pushes the isolation burden back onto your own code: nothing at the AWS API level stops one tenant's leftover files or environment variables from being visible to whichever tenant is routed to the same instance next. You own that boundary now, not Firecracker.

Most production systems land somewhere in between: dedicated MicroVMs for tenants who pay for it or whose workload demands it (long-running sessions, compliance requirements, workloads that install arbitrary packages per Chapter 8), and a shared pool for everyone else. The example below models exactly this split.

🏭 In production

"Per-user isolation" in AWS's own use-case list is a real, intentional feature of the service, not something you're bolting on. But the service gives you the primitive (a MicroVM you can dedicate to one tenant); the policy of which tenants get a dedicated instance is yours to write, and that's most of this chapter.

Session Management and Routing

Whichever tier a tenant lands in, something in front of RunMicrovm has to decide, for every incoming request: does this tenant already have an assigned MicroVM, or does one need to be created (or reused from a pool)? That decision is session routing, and it sits above the AWS API entirely. RunMicrovm, GetMicrovm, and the rest (Chapters 5–6) don't know anything about tenants, only about MicroVM identifiers.

A minimal router needs three pieces of state: which tenants are dedicated and which MicroVM they own, which shared-pool slots exist and who currently owns each one, and an eviction order for the shared pool so that when it's full, the router has a deterministic answer for who gets bumped. Second, the shared pool with only two slots starts evicting almost immediately once a third distinct tenant shows up, and every eviction is logged as a cleanup event, which is the subject of the next section.

⚠️ Gotcha

A router that calls RunMicrovm synchronously on the request path adds real latency. Chapter 3 noted Firecracker's own boot time is under 125ms, but image/network setup on top of that was closer to a few seconds in the live transcript captured for Chapters 5–6. For dedicated tenants this cost is paid once, at first use. For a shared pool that's constantly evicting and reassigning slots under load, that per-request cost becomes the norm, not the exception. That's another argument for sizing the pool to your actual concurrent-tenant count rather than making it as small as possible to save cost.

Cost vs. Security Tradeoffs

The dedicated/shared decision is fundamentally a cost-vs-isolation tradeoff, and it's worth being explicit about what you're buying on each side rather than picking one tier for the whole system by default.

  • Dedicated MicroVMs cost more to keep warm. Even with an aggressive maxIdleDurationSeconds (Chapter 6) suspending an idle tenant's MicroVM, you are still paying for a distinct instance, a distinct endpoint, and distinct snapshot storage per tenant. At scale, meaning thousands of low-activity tenants, that adds up even with suspension helping.
  • Dedicated MicroVMs give you the isolation guarantee for free. You don't have to write cleanup code, audit it, or worry about a bug in that cleanup code leaking one tenant's data to the next, because there is no "next" on the same instance.
  • Shared pools cost less per tenant because a fixed, small number of MicroVMs serves an arbitrarily larger number of low-activity tenants. This is the same economic argument as connection pooling in a database driver, applied to compute instead of connections.
  • Shared pools push isolation risk onto your own code. Every bug in the cleanup step covered in the next section is a potential cross-tenant data leak. That is a materially different risk profile than the dedicated tier's "Firecracker's boundary handles it."

A reasonable default: dedicated for any tenant whose workload involves secrets, long-lived state, or a compliance requirement that specifically names per-tenant isolation; shared pool for short, stateless, low-trust requests (a public code-execution playground, a free tier with rate limits) where the isolation given up is a request-level VM boundary, not a data-confidentiality boundary, and where you have already applied the request-level defenses from Chapter 10 regardless of tier.

Cleaning State Between Users

If you choose a shared pool for any tier of tenant, "clean the instance between users" is not optional polish. It is the entire isolation mechanism for that tier, and it has to happen on every single reassignment, not just "usually." Chapter 9 covered cleaning a single tenant's own leftover files between their own requests; this is the same problem at the tenant boundary, where the stakes are cross-customer data exposure rather than one user's own disk filling up.

What "clean" has to mean in practice, on a shared MicroVM:

  • Filesystem. Any file the previous tenant wrote into a shared working directory. Chapter 9's approach (wipe a scoped working directory rather than trying to enumerate and delete individual files) applies directly here.
  • Process state. Kill any lingering child processes, background jobs, or long-running servers the previous tenant started. A shared MicroVM that reuses a Python interpreter process across tenants is especially dangerous here: module-level globals, open file handles, and cached imports can all carry state across the boundary.
  • Environment variables and secrets. If the previous tenant's request injected an API key or credential into the guest's environment, that has to be scrubbed before the next tenant's code can run, or their process can read it.
  • Network connections. Any outbound connection the previous tenant's code opened (Chapter 7) should be closed, not left dangling for the next tenant to inherit or observe.

The example's reset_workspace() call is a placeholder for all of this. In the simulation it does nothing, deliberately, because the point of the example is the routing decision (when a slot changes owners), not a real filesystem wipe. A real implementation would issue an actual cleanup command inside the guest (or, more reliably, snapshot-and-restore the guest to a known-clean image state, since an in-place wipe can miss something a fresh restore cannot).

🔬 Under the hood

The most reliable version of "clean state between users" on a shared instance isn't a cleanup script at all. It's discarding the running MicroVM and launching a fresh one from the image's snapshot (Chapter 3's suspend/resume model makes this cheap relative to a cold boot). A cleanup script has to enumerate every category of state that could leak; a fresh snapshot restore starts from the same known-good state every time by construction. The tradeoff is the latency noted above: a fresh instance costs more than reusing a warm one.

IAM Roles and Permissions per Tenant

Every RunMicrovm call in the transcripts used throughout this course (Chapters 5–6) has passed an --execution-role-arn: an IAM role the MicroVM assumes for whatever AWS calls it makes from inside the guest. In a single-tenant sandbox that role is usually one fixed, broad-enough role. In a multi-tenant sandbox, it's the mechanism for making sure tenant A's MicroVM cannot touch tenant B's S3 objects, Secrets Manager entries, or any other AWS resource, even if the application-level routing logic has a bug.

The account used to capture this course's live transcripts has, among others, MicroVMLambdaPOCRole (the general execution role used in every Chapter 5–6 example) and LambdaMicroVMsNetworkConnectorRole. A single-tenant course example can reuse one role for everything. A real multi-tenant system should not: the execution role passed to RunMicrovm is exactly where per-tenant AWS-level scoping belongs, and it should be scoped with the same discipline you'd apply to any other per-customer IAM boundary: a distinct role (or a role assumed with a tenant-specific session tag and a matching resource policy) per tenant, so that even a routing bug that sends tenant A's request to tenant B's dedicated MicroVM cannot escalate into tenant A's code touching tenant B's AWS resources through that MicroVM's role.

Two patterns show up in practice, in increasing order of operational cost and decreasing order of blast radius if something goes wrong:

  1. One execution role per tenant. Simple to reason about, but role creation and management scales linearly with tenant count: fine for hundreds of dedicated enterprise tenants, unwieldy for a shared pool serving thousands of low-trust users.
  2. One execution role, scoped by tenant ID at assume-role time. A single role's trust policy allows your orchestrator to assume it with a session tag identifying the tenant, and the role's own permission policy uses that tag in its resource conditions (the same session tags pattern AWS documents generally for scoping assumed-role access by attribute). This scales to a shared pool without a role-per-tenant explosion, at the cost of a more complex policy to get right and audit.

Whichever pattern you pick, it composes with everything else in this chapter rather than replacing it: IAM scoping bounds what a tenant's MicroVM can do to AWS resources outside the VM; the dedicated-vs-shared decision and the cleanup discipline bound what one tenant can see inside the VM boundary. Neither one substitutes for the other.

Challenge: Build a Multi-User Sandbox

Your challenge is to implement the routing logic behind a two-tier multi-tenant sandbox: dedicated MicroVMs for tenants who need them, and a fixed-size shared pool for everyone else, with correct eviction and cleanup bookkeeping when the pool is full.

Open challenges/ch11.py and implement IsolationRouter:

  • route(tenant_id, tier) — for tier="dedicated", create a MicroVM on first use and return the same one on every later call for that tenant. For tier="shared", reuse a tenant's existing slot if they have one; otherwise use an empty slot if the pool isn't full; otherwise evict the least-recently-assigned slot, report cleaned=True, and assign it to the new tenant.
  • evict_dedicated(tenant_id) — remove a dedicated tenant's assignment entirely (simulating TerminateMicrovm), returning whether there was one to remove.

Run the tests:

cd challenges && python3 -m pytest ch11_test.py -v

Thirteen tests cover both tiers: dedicated reuse and eviction, shared-pool slot reuse, filling an empty pool before evicting anything, least-recently-assigned eviction order, and the cleaned flag on reassignment. See solutions/ch11.py when you're ready to compare.
