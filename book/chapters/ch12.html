Chapter 12

Security Hardening

A MicroVM gives you a strong isolation boundary by default. Hardening a sandbox is the discipline of not undoing that boundary at every layer above it.

Network Restrictions and Egress Control

Chapter 7 introduced the two network connectors a MicroVM launches with: HTTP_INGRESS and INTERNET_EGRESS, both attached by default unless you override them with --ingress-network-connectors/--egress-network-connectors on RunMicrovm. That default is convenient for getting a sandbox working, and wrong for a hardened one: untrusted code with unrestricted internet egress can exfiltrate data, reach an attacker's command-and-control endpoint, or use your MicroVM as a relay for outbound abuse, and none of that requires it to escape the VM at all — it just has to make an HTTP request.

Hardening egress means treating "no outbound network access" as the default and adding back only what a given sandbox's workload actually needs, the reverse of the service's own out-of-the-box default. A code-execution sandbox for a data-analysis agent might need pip/PyPI access (Chapter 8) and nothing else; a general-purpose coding assistant sandbox might need package registries and a handful of API endpoints the agent is explicitly allowed to call. Neither needs unrestricted internet access, and granting it by default is the single easiest way to turn a contained code-execution bug into a data exfiltration incident.

The same discipline applies to self-hosted Firecracker deployments that don't have Lambda's managed network connectors to lean on. A forward proxy in front of the guest's only route out — allow-listing destination hosts rather than trying to block a deny-list of bad ones — is the standard pattern: a deny-list has to anticipate every bad destination in advance, while an allow-list fails closed by construction. A Squid proxy configured with an explicit http_access allow allow-list (and everything else implicitly denied) is a common, well-understood way to implement this without touching the guest's own iptables rules, which the guest's untrusted code could otherwise tamper with if it ever got enough privilege inside the VM to try.

⚠️ Gotcha

Blocking egress at the network layer does not block DNS-based exfiltration by itself — a process that can resolve arbitrary hostnames can smuggle small amounts of data out one DNS query at a time, even with HTTP/HTTPS egress fully blocked. An allow-list has to constrain DNS resolution too (only resolve to your allowed destinations, or route DNS through the same proxy that enforces the HTTP allow-list), not just the transport-layer connection.

Dangerous Operations to Block

Some operations are worth blocking regardless of what the sandbox's workload is nominally supposed to do, because no legitimate use of an untrusted-code sandbox needs them and every one of them is a documented building block of a sandbox escape or a denial-of-service against the host:

  • Shelling out with an interpreted shell string — os.system, subprocess.run(..., shell=True), and equivalents hand a string straight to /bin/sh, which turns any string-formatting bug in your own code into arbitrary command execution, not just whatever command you intended to run.
  • Raw syscall or FFI access — ctypes, cffi, and similar let code call arbitrary C functions and make raw syscalls directly, sidestepping any restriction you've placed at the Python level entirely. Chapter 7's subprocess.run timeout, for instance, has no effect on code that never goes through subprocess at all.
  • Fork bombs and uncontrolled process creation — os.fork() in a loop (or any unbounded process-spawning pattern) can exhaust a host's process table before a wall-clock timeout even fires. This is exactly the class of attack Chapter 10's resource limits and the jailer's cgroup controls (below) exist to contain at the OS level, but rejecting the pattern before execution is cheaper than relying on the limit to kick in.
  • Raw socket access — socket.socket() called directly, rather than through an HTTP client that respects your egress allow-list, is a way to route around the allow-list rather than through it.
  • eval/exec of untrusted strings — running a string as code inside code that is already untrusted collapses whatever review or scanning you did on the original snippet, since the actual behavior is decided at runtime from data your scanner never saw.

None of this is a complete list, and it isn't meant to be — it's the small set of patterns worth blocking unconditionally, before you even get to whether a particular sandbox's workload needs broader restrictions on top.

Validating User Code Before Execution

Every defense discussed so far in this book — the MicroVM boundary itself, egress restrictions, resource limits — operates once code is already running inside the guest. There's a cheaper layer available before that: reject obviously dangerous code before it is ever handed to RunMicrovm at all. This is squarely an application-level control, and it is deliberately not a replacement for anything below it in this chapter — a static scanner over source text can be defeated by encoding a payload as a base64 string and decoding it at runtime, or by building a dangerous call out of getattr indirection. What it buys you is cheap: rejecting an obviously malicious snippet in milliseconds, without ever paying for a MicroVM launch, and without relying on the OS-level layers to be the only thing standing between untrusted code and the host.

The example below is a static scanner built on Python's own ast module rather than regular expressions over source text — parsing the code into a syntax tree means os . system(...) with extra whitespace, or a call written as getattr(os, "sys" + "tem"), is caught by structure rather than missed by a brittle string match (the getattr-indirection case specifically is still not caught here, which is exactly the kind of gap worth naming rather than glossing over). It walks the tree looking for three things: dangerous imports (ctypes, cffi), calls to a fixed set of dangerous fully-qualified names (os.system, subprocess.run, socket.socket, and the rest of the previous section's list), and any call carrying a shell=True keyword argument regardless of which function it's attached to.

Notice the shell=True sample produces two separate violations — one for the call target (subprocess.run is itself blocked, since it can spawn arbitrary processes even without a shell) and one for the keyword argument. That overlap is intentional: the keyword check exists precisely so that a wrapper function hiding the real call target still gets caught by the shell=True pattern even if the call target itself weren't already on the list.

🏭 In production

Treat a static scanner as a fast-reject filter, not a security boundary. It exists to save the cost of a MicroVM launch on code that's obviously hostile and to give a user immediate feedback ("this snippet uses a blocked import") instead of a confusing runtime failure inside the sandbox. The MicroVM boundary, seccomp filters, and jailer isolation below are what actually contain code that gets past the scanner — deliberately or not.

Monitoring for Malicious Behavior

Validation before execution catches what you can recognize in the source text. Monitoring during and after execution catches what you can't — behavior that only shows up once code is actually running, whether because it was obfuscated past the scanner or because the behavior in question isn't a static pattern at all (an unusually long-running process, an unexpectedly large amount of outbound data, a spike in CPU that doesn't match the workload).

Three signals are worth watching for every execution, building directly on infrastructure earlier chapters already introduced:

  • Resource usage anomalies. Chapter 10's per-execution CPU/memory/disk limits stop a single runaway process from exhausting the host, but the limits themselves are also a monitoring signal: a sandbox that repeatedly hits its memory ceiling, or one whose CPU usage pattern looks like a mining workload rather than the agent task it was launched for, is worth flagging even though the limit itself already contained the immediate damage.
  • Egress volume and destination anomalies. If the egress allow-list from earlier in this chapter is in place, an execution that tries to reach a host outside that list is itself a monitoring event, not just a blocked connection — a legitimate workload doesn't normally probe for hosts it isn't allowed to reach, and a pattern of repeated denied attempts is a stronger signal than any single one.
  • Execution duration outliers. A code snippet that normally finishes in under a second and is now running for minutes is worth surfacing even before Chapter 10's hard timeout fires, since the timeout is a backstop, not an early-warning system.

Chapter 13 covers the CloudWatch integration and logging patterns that make this practical to build; this chapter's job is naming what's worth watching for, since the plumbing to watch it is a separate concern from deciding what "malicious" looks like for your workload.

Understanding Firecracker's Security Layers

Everything above this section happens in your application, above the MicroVM boundary. This section is about what's underneath that boundary — the layers Firecracker itself provides, which apply whether or not your application-level scanner catches everything. Chapter 2 introduced these briefly; here's the security-specific detail on each.

KVM: hardware-enforced isolation

Firecracker's own documentation states the foundation plainly: "Firecracker microVMs use KVM-based virtualizations that provide enhanced security over traditional VMs." KVM means the guest is running behind actual CPU-level virtualization (VT-x/AMD-V), with its own kernel and its own address space, separated from the host by hardware privilege levels rather than by a Linux namespace or cgroup. That's a categorically different boundary than a container: escaping it requires a hypervisor-level vulnerability, not just a kernel bug reachable from an unprivileged process, which is why it's the layer everything else in this chapter is built on top of rather than a replacement for.

Seccomp: restricting what the VMM itself can do

KVM isolates the guest from the host. Seccomp restricts what Firecracker's own host-side process (the VMM) is allowed to do, in case an attacker somehow escapes the guest and lands in that process. Firecracker installs thread-specific seccomp-BPF filters that allow-list the specific syscalls each of its threads needs and deny everything else — the same allow-list-over-deny-list logic as the egress proxy earlier in this chapter, applied to the syscall table instead of the network. An attacker who breaks out of the guest doesn't get a general-purpose host process to work with; they get a process that can only make the small set of syscalls Firecracker itself needs to run a microVM.

Jailer: a second line of defense

The jailer is a separate companion program that wraps the Firecracker process with conventional Linux sandboxing, on top of the virtualization boundary and the seccomp filters. Per the jailer's own documentation, it "is a program designed to isolate the Firecracker process in order to enhance Firecracker's security posture" and is meant to address Firecracker's security needs specifically — not a general-purpose sandboxing tool. Concretely, before the Firecracker binary is ever exec'd, the jailer:

  • Builds a chroot jail via pivot_root(), so the Firecracker process's filesystem view is limited to a directory tree the jailer constructed, not the host's real root.
  • Creates dedicated mount, PID, and (optionally, via --new-pid-ns) network namespaces.
  • Applies cgroup resource limits — --cgroup for arbitrary cgroup controller values, plus a dedicated --resource-limit flag for things like open file descriptor counts and file size, using the same cgroups v1/v2 mechanism Firecracker's production host setup guide documents for per-instance memory (memory.limit_in_bytes), CPU (cpu.cfs_quota_us), and disk I/O (blkio.throttle.io_serviced) controls — the same primitives Chapter 7's under-the-hood note referenced for resource limiting.
  • Drops privileges to a specified non-root --uid/--gid before finally exec'ing Firecracker, so the running VMM process never holds more privilege on the host than it was explicitly given.

Firecracker's documentation frames the relationship between these layers directly: each microVM is "further isolated with common Linux user-space security barriers by a companion program called 'jailer,'" which "provides a second line of defense in case the virtualization barrier is ever compromised." That's the whole point of defense in depth here — KVM is the primary wall, seccomp narrows what a host-side escape can do, and the jailer's namespaces/chroot/cgroups/uid-drop are a second, independent wall that doesn't depend on the first one holding.

🔬 Under the hood

Lambda MicroVMs runs on this same Firecracker foundation, but you never invoke the jailer yourself — AWS operates it as part of the managed service, the same way you never see the seccomp filter files or the chroot setup. Understanding these layers matters for this course anyway, for two reasons: it's what "MicroVM-level isolation" concretely means when Chapter 1 first claimed it was stronger than a container, and it's exactly what you'd need to configure yourself if Chapter 17's self-hosted Firecracker path is ever the right call for a workload Lambda MicroVMs doesn't fit.

Challenge: Harden Your Sandbox Against Common Attacks

Your challenge is to build the application-level scanner described earlier in this chapter: a validator that inspects Python source before it is ever handed to a MicroVM, and rejects the patterns this chapter named as unconditionally dangerous.

Open challenges/ch12.py and implement SandboxHardener.scan():

  • Block imports of ctypes or cffi (raw FFI/syscall access), in both import and from ... import form.
  • Block calls to os.system, os.popen, os.fork, os.execv/execve/execl, subprocess.Popen/call/run/check_output, and socket.socket.
  • Block any call carrying a shell=True keyword argument, regardless of which function it's attached to.
  • Block the builtins eval, exec, and __import__.
  • Treat a snippet that fails to parse as unsafe — code you can't parse is code you can't reason about.
  • Leave ordinary code (arithmetic, comprehensions, safe stdlib use like json) alone: it must come back safe=True with no violations.

Run the tests:

cd challenges && python3 -m pytest ch12_test.py -v

Fifteen tests cover safe code passing cleanly, each dangerous import/call/builtin being caught, the shell=True keyword check firing even when attached to an otherwise-unlisted call, and syntax errors being treated as unsafe rather than silently allowed. See solutions/ch12.py when you're ready to compare.

⚡ Try it

Run examples/ch12_code_scanner.py to see the reference scanner classify a mix of safe and deliberately dangerous snippets — fork bombs, raw ctypes access, eval of untrusted input, and more — entirely offline, with no MicroVM launch required.
