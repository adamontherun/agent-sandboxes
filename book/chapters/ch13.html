Chapter 13

Observability and Debugging

A sandbox you can't see into is a sandbox you can't fix. Observability turns a black-box MicroVM into something you can reason about after the fact, without weakening the isolation boundary that makes it useful in the first place.

CloudWatch Integration for Logs and Metrics

Lambda MicroVMs streams both build-time and runtime logs to CloudWatch Logs. The integration is configured through the logging parameter on CreateMicrovmImage, which the API documentation describes as:

"The logging configuration for build-time and runtime logs. Specify {"cloudWatch": {"logGroup": "..."}} to stream logs to a custom CloudWatch log group, or {"disabled": {}} to turn off logging."

In practice, if you supply a logGroup name (or let the service use its default), Lambda MicroVMs creates one log group per image name under the prefix /aws/lambda-microvms/. Within that group, log streams follow two naming patterns observed in real deployments:

  • <version>/<request-uuid> — build-time and early-runtime streams (e.g. 11.0/0f34abe4-307a-4882-bb74-c34595e822cf)
  • <date>[<version>]microvm-<microvm-id> — longer-running session streams (e.g. 2026/06/24[11.0]microvm-b510854e-5a9e-3da7-b55c-78980805c501)

⚠️ Gotcha

There is no dedicated CloudWatch metrics namespace for Lambda MicroVMs (no AWS/LambdaMicroVMs dimension in CloudWatch Metrics). If you need quantitative signals (error rates, p99 execution duration, executions per hour), you build them from log data using CloudWatch Logs Insights queries, or by polling GetMicrovm/ListMicrovms for state-based signals. Don't go looking for pre-built dashboards or metric alarms the way you would for Lambda Functions; they don't exist for this service today.

Tracing Execution Flow

CloudWatch Logs captures whatever your application writes to stdout/stderr. That means structured logging is entirely your responsibility. There's no X-Ray integration or distributed tracing built into the service. The practical pattern is: emit one JSON object per log line, with a consistent request_id field across all events belonging to the same execution, so you can correlate them later in Logs Insights without parsing unstructured text.

The minimal set of fields that makes a log line useful after the fact:

  • timestamp — ISO 8601, so Logs Insights' @timestamp field and your own are both usable without conversion.
  • request_id — a UUID generated at the start of each execution, carried through start and end events.
  • event — a dot-namespaced string (execution.start, execution.end, health_check) you can filter on.
  • duration_ms — on end events, how long the execution took.
  • status — success, error, or timeout.

Each pair of start/end lines shares a request_id. In CloudWatch Logs Insights, you'd query them with fields @timestamp, request_id, event, status, duration_ms | filter event like /execution/ | sort @timestamp desc to get a table of every execution's outcome without opening individual streams.

🏭 In production

For high-traffic deployments, create a CloudWatch Logs metric filter on patterns like "Exception" or "error" to drive a CloudWatch Alarm without running Insights queries on a schedule. Metric filters are evaluated on ingestion (no query cost), so they give you near-real-time alerting on failure rates as a free side effect of logs you're already paying to store.

Cost Monitoring and Optimization

The primary cost lever Lambda MicroVMs exposes is the idle policy. The developer guide describes suspension as a way of "preserving memory and disk state while reducing costs" when a MicroVM sits idle, and separately notes that pricing works on a baseline-plus-burst model: "you pay the baseline rate while your MicroVM is running and only pay for active use above the baseline."

The three fields that control this are the same ones Chapter 6 introduced for lifecycle management, now viewed through a cost lens:

  • maxIdleDurationSeconds — how long a MicroVM sits idle before auto-suspending. Lower values save money on instances that are mostly waiting; higher values avoid resume latency for bursty workloads.
  • suspendedDurationSeconds — how long a suspended instance is kept before being terminated entirely. Suspended instances cost less than running ones but more than nothing (state is preserved on disk).
  • autoResumeEnabled — whether an incoming request automatically wakes a suspended instance. If false, suspended instances must be explicitly resumed via ResumeMicrovm.

There is no dedicated Cost Explorer or Budgets integration specific to Lambda MicroVMs. Cost visibility comes from the same sources as any other AWS service: the billing console's service-level breakdown and (if you need finer granularity) cost allocation tags applied to your MicroVM images. The observability-specific angle is: if you're running more MicroVMs than you need, or leaving them running longer than they're used, the signal is in your own logs and ListMicrovms output. Count how many are in RUNNING state versus how many had an execution in the last N minutes, and you've built the only "cost metric" the service gives you the raw data for.

🔬 Under the hood

Aggressive auto-suspend (e.g. maxIdleDurationSeconds: 60) is the simplest cost optimization for sandbox workloads where executions are bursty and separated by minutes or hours of idle time. The tradeoff is resume latency, which isn't a number AWS documents directly and will vary with memory size and workload. If your workload is latency-sensitive and bursty, measure resume times in your own logs before tightening the idle threshold.

Health Checks and Error Handling

The cheapest health signal for a MicroVM is the state and stateReason fields on a GetMicrovm response. Polling this is faster and cheaper than running a Logs Insights query, and it tells you whether an instance is alive without having to parse log content:

A health-check function maps these states to actions:

  • RUNNING — healthy, no action needed.
  • SUSPENDED — healthy (cost-optimized idle state), will auto-resume on next request if autoResumeEnabled is true.
  • TERMINATED — investigate. Check stateReason: "Success." means a clean shutdown (e.g. maximumDurationInSeconds reached after 8 hours, or explicit TerminateMicrovm call); anything else is unexpected.
  • FAILED or unknown — investigate immediately. Check CloudWatch Logs for the corresponding stream.

For error handling in your orchestrator, the pattern is: attempt the execution, check the HTTP response from the MicroVM's endpoint, and if it fails, check GetMicrovm to determine whether the failure is transient (instance still running, just a code error) or structural (instance terminated/failed, needs a new RunMicrovm). Log both the execution-level error and the instance state so you can correlate them later.
