Chapter 14

Production Patterns

Everything in Part IV so far has been about one MicroVM behaving correctly under load, attack, or scrutiny. This chapter is about a fleet of them behaving correctly over time — new image versions, more traffic than one instance can serve, launches that fail for reasons that have nothing to do with your code, and sessions that eventually just... end.

Image Versioning and Updates

Every CreateMicrovmImage call doesn't overwrite the previous image — it produces a new, numbered version. Chapter 13's research turned up a real example of this from a live account: the lambda-microvms-poc-hello-world image sits at version 11.0, and ListMicrovmImageBuilds confirmed that a single image version can have more than one build behind it, one per Graviton chipset generation. A production orchestrator has to think in terms of "which version is a given running MicroVM on," not just "which image."

RunMicrovm's real response (captured live against that same account in Chapter 5's research) includes the field directly:

Track that field the same way you'd track a container image digest: log it on every launch, and keep a small mapping in your orchestrator's own state (a database row, a config value — not necessarily anything fancy) of which version is currently considered "live" for new traffic. UpdateMicrovmImage and UpdateMicrovmImageVersion exist for changing an image's configuration after the fact, but neither one silently migrates already-running MicroVMs to a new version — a running instance keeps the version it launched with until it's terminated and relaunched. That's the property the next section builds on.

⚠️ Gotcha

ListMicrovmImageBuilds requires --image-version as a required parameter, not just an image identifier — you can't list every build across every version of an image in one call. If your orchestrator needs a full build history, iterate ListMicrovmImageVersions first, then call ListMicrovmImageBuilds once per version.

Blue/Green Deployment for Sandbox Updates

Because a running MicroVM never migrates to a new image version on its own, rolling out a new version is naturally a launch-new-then-retire-old operation, not an in-place upgrade. That maps directly onto blue/green deployment: "blue" is whatever image version is currently taking new sessions, "green" is the candidate you just built. The only rule that matters is: never flip live traffic to green until you've launched it and confirmed it's actually healthy.

"Healthy" here reuses Chapter 13's health-check logic exactly — a MicroVM in RUNNING or SUSPENDED state is fine to serve traffic; TERMINATED or FAILED means something went wrong during launch and the cutover should abort, leaving blue as the live version. This is the same GetMicrovm state machine Chapter 13 introduced for debugging, now used as a deployment gate instead of a debugging aid.

Notice the second cutover attempt: the candidate's launch itself succeeded well enough to return a state, but that state was TERMINATED, and the orchestrator correctly refuses to move live traffic onto it. That's the entire point of gating on health rather than on "did the API call return 200" — a MicroVM can launch and still not be something you want serving requests.

🏭 In production

Keep at least one blue instance warm during a cutover window rather than terminating it the instant green passes its health check. Sessions that were mid-execution on blue when the cutover happened should be allowed to finish rather than getting cut off — drain blue on a timer (or once its active-session count hits zero), the same pattern you'd use for any stateful service, not just MicroVMs.

Scaling: When to Launch More MicroVMs

Lambda MicroVMs has no autoscaling group, no target-tracking policy, and (per Chapter 13's research) no dedicated CloudWatch metrics namespace to drive one even if it did. The scaling signal is whatever your own orchestrator computes from ListMicrovms output and its own request queue — there is nothing built into the service to delegate this to.

A minimal heuristic that's good enough for a bursty sandbox workload: launch another MicroVM if there are zero running and requests are queued, or if the backlog per running instance crosses a threshold; scale down if instances are idle with no queue. The example script's decide_scaling_action implements exactly this and the real output above shows all four branches: launch when nothing's running yet, launch when the backlog per instance is too deep, scale_down when instances sit idle, and hold when things are balanced.

Whichever heuristic you pick, remember Chapter 11's framing still applies underneath it: "launch more MicroVMs" and "route more sessions onto fewer shared MicroVMs" are two different scaling strategies with the same cost/isolation tradeoff that chapter covered for multi-tenancy, not two names for the same thing.

Error Recovery and Retry Logic

Not every launch failure means the same thing. Chapter 13 already drew this distinction for execution-level errors: check whether a failure is transient (the instance is fine, the specific request failed) or structural (the instance itself is gone or never came up). The same split applies one level up, at the launch call itself.

A throttling or capacity error from RunMicrovm is transient — retrying with backoff is the right response, and Chapter 5's research already surfaced a real, easy-to-misread ParamValidation error (a missing idlePolicy field) that looks like it needs a retry but actually needs a code fix. Distinguishing those two cases matters: retrying a malformed request forever just wastes time and API calls. The example orchestrator's launch_with_retry only retries on a distinct TransientLaunchError, and lets anything else propagate immediately, for exactly this reason.

Backoff itself should be exponential and capped, with a little jitter, so that a fleet of orchestrator workers hitting the same throttle don't all retry in lockstep and re-trigger the same throttle a second later. The example script's backoff (min(2 ** (attempt - 1), 8) seconds plus a small random jitter) is a standard shape for this — nothing MicroVM-specific, the same pattern you'd use retrying any rate-limited AWS API call.

🔬 Under the hood

The AWS CLI itself already retries transient errors on most API calls by default (adaptive retry mode with exponential backoff), so if you're calling RunMicrovm through the CLI or an SDK rather than raw HTTP, you may already be getting some of this for free at the transport level. Application-level retry logic like this chapter's example still matters on top of that for anything the SDK doesn't retry automatically — a launch that "succeeds" into a bad state, for instance, which no HTTP-layer retry logic can detect.

Designing for the 8-Hour Limit

Chapter 10 introduced the hard ceiling: maximumDurationInSeconds defaults to 28800 seconds, which is exactly 8 hours, confirmed on a real RunMicrovm response in this course's own research. Chapter 13 then showed what a clean expiry actually looks like from the outside — a real terminated instance with "stateReason": "Success.", indistinguishable at that field from an explicit TerminateMicrovm call. A production orchestrator has to plan for that termination before it happens, not just handle it afterward.

The practical pattern is to track each instance's startedAt timestamp (also present on the real RunMicrovm response) and treat "approaching 8 hours" as its own health signal, distinct from the RUNNING/SUSPENDED/TERMINATED state machine Chapter 13 built. A long-running agent session that's going to need more than 8 hours of wall-clock time isn't a bug to work around inside one MicroVM — it's a signal to checkpoint whatever state matters (the same persistent-disk state Chapter 3 and Chapter 9 covered) and hand off to a freshly launched instance before the clock runs out, rather than discovering the hard way that the service terminated a session mid-task.

Note that maxIdleDurationSeconds (Chapter 6/13's auto-suspend knob) and maximumDurationInSeconds are independent limits: a MicroVM that's actively handling requests can still be auto-suspended for being idle between them, and separately still hits the hard 8-hour ceiling regardless of how much of that time it spent suspended versus running. Neither one substitutes for the other in your orchestrator's bookkeeping.

Challenge: Build a Production-Ready Orchestrator

Your challenge is to implement the orchestrator this chapter walked through: image-version tracking, retry-with-backoff around a flaky launch call, health-gated blue/green cutover, and the scaling heuristic.

Open challenges/ch14.py and implement ProductionOrchestrator:

  • __init__(live_version) — sets the initially live image version; no candidate staged yet.
  • stage_candidate(version) — records a pending candidate version.
  • launch_with_retry(launch_fn, image_version, max_attempts=4) — calls launch_fn, retrying on TransientLaunchError up to max_attempts times, recording every attempt (success or failure) in self.attempts; raises RuntimeError if every attempt fails.
  • cut_over(instance) — health-checks a launched instance dict (RUNNING/SUSPENDED are healthy) and only then flips live_version to the staged candidate; raises ValueError with no candidate staged.
  • decide_scaling_action(running_count, queued_requests, max_per_microvm=5) — returns "launch", "scale_down", or "hold" per the heuristic above.

Run the tests:

cd challenges && python3 -m pytest ch14_test.py -v

Seventeen tests cover candidate staging, retry accounting and exhaustion, cutover health-gating for each MicroVM state, and every branch of the scaling heuristic. See solutions/ch14.py when you're ready to compare.
