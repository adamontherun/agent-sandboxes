Chapter 15

AI Coding Assistant Pattern

Everything so far has been about the sandbox as infrastructure: launching it, securing it, watching it, keeping it running. This chapter looks at the specific shape an AI coding assistant asks that infrastructure to take — a persistent workspace an LLM drives turn by turn, writing files, running commands, and reading the results back, for as long as a development session lasts.

How Tools Like Claude Code and Cursor Use Sandboxes

AWS frames this directly as one of the use cases Lambda MicroVMs was built for. The service's own product page lists "AI-powered development tools generate and execute code on behalf of users" under an "AI coding assistants and agent sandboxes" section, and describes the value proposition as "a separate execution boundary per task: an isolated compute environment with no access to agent state and no shared state across users." The product page doesn't name specific products, and neither does the launch announcement, which groups "AI coding assistants" alongside "interactive code environments, data analytics platforms, vulnerability scanners, and game servers that run user-supplied scripts" as instances of the same underlying pattern: something needs to run arbitrary, possibly untrusted code on a user's behalf, in an environment isolated from every other user's.

What does get documented in public, specifically, is how Claude Code isolates the shell commands it runs locally. Claude Code's own sandboxing docs describe a Bash-tool sandbox built into the CLI that enforces filesystem and network boundaries at the OS level — Seatbelt on macOS, bubblewrap on Linux and WSL2 — restricting writes to the working directory and a session temp directory by default, and routing network access through a proxy that only allows domains you've approved. That's a real, citable architecture, but it's worth being precise about what it is: OS-level process sandboxing on the machine Claude Code is already running on, not a MicroVM. It solves a different part of the same problem this book has been building toward — keeping a shell command from touching more of the filesystem or network than it needs to — using primitives closer to a container boundary than a virtual machine.

⚠️ Gotcha

Don't conflate "Claude Code has a documented local sandbox" with "Claude Code's cloud execution runs on Lambda MicroVMs specifically." The two facts are both true and both citable, but they answer different questions: one is about a CLI process's isolation on your machine, the other is about which infrastructure a hosted execution environment happens to use. No public source in this book's research confirms the internal infrastructure choice behind any specific hosted coding-assistant product's remote execution feature (Claude Code's own remote/cloud environments, Cursor's background agents, or similar). That's a real gap in what's publicly documented, not a detail this book is choosing to omit.

What's genuinely well-documented, and what carries forward into the rest of this chapter, is the general pattern one level up from any single vendor's implementation: a coding assistant sandbox generally needs (1) a persistent workspace that outlives any single tool call, (2) a way to execute commands and capture their output in a form the LLM can read, and (3) some boundary limiting what that execution can touch on the host or the network. Chapters 7, 9, 11, and 13 already built each of those pieces against the real Lambda MicroVMs API: code execution, filesystem persistence, per-tenant isolation, and observability. This chapter's job is showing how they compose into one loop, not re-deriving any of them from scratch.

Read-Eval-Print-Loop in a MicroVM

An AI coding assistant's core loop looks a lot like a REPL, except the thing typing at the prompt is an LLM instead of a human: write a file, run a command, read the output, decide what to do next, repeat. Each of those steps is a tool call the model issues and a result the sandbox returns.

examples/ch15_repl_session.py demonstrates this loop against a Session object. To be explicit about what this is: this is a local simulation of a MicroVM boundary, not an actual MicroVM. Session wraps an ordinary temp directory on this machine, and run_command shells out via subprocess scoped to that directory — there's no RunMicrovm call, no Firecracker, no real VM isolation anywhere in this script. The point is to show the shape of the loop cheaply, without an AWS account, before wiring the same shape onto the real API calls Chapters 4-6 covered.

Running the script drives a small multi-turn session — write a script with a bug, run it, watch it fail, fix it, run it again, then read the file back to confirm what's actually on disk. This is real, captured output from actually running the script, not an invented transcript:

Notice Turn 2's failure never breaks the loop — the assistant sees the traceback as a normal tool result, the same way it sees success, and decides what to do next from that. That's the entire point of shaping results as structured data (ToolResult's ok/output/detail fields) rather than letting a failing command raise an exception the caller has to catch: a REPL loop driven by an LLM needs every outcome, including failures, to come back as something the model can read and reason about in its next turn.

Turn 6 is the one line in _resolve() standing in for a real sandbox's filesystem boundary: it rejects any path that would resolve outside the session root, the same property Chapter 9 covered for a real MicroVM's filesystem isolation, just enforced here in a single Python check instead of an OS-level mount boundary. On a real MicroVM, that boundary would instead be the isolated filesystem Chapter 9 described, not application code checking path prefixes — this simulation substitutes the cheaper check because there's no real VM boundary to lean on here.

Managing Long-Running Development Sessions

A single coding-assistant conversation can span dozens or hundreds of tool calls over an extended period — not one request-response cycle, but a session that keeps a workspace alive across many turns while the user (or the model, working autonomously) iterates. Every constraint Chapter 14 covered for a production orchestrator applies directly here, just from the session's point of view instead of the fleet's:

  • The 8-hour maximumDurationInSeconds ceiling (confirmed on a real RunMicrovm response in this course's own research, see Chapter 5) is a hard stop a long development session can actually hit. A session that runs long enough needs the same checkpoint-and-handoff pattern Chapter 14 described for any long-running MicroVM workload: persist what matters, then launch a fresh instance rather than discovering mid-task that the old one was terminated.
  • Chapter 6 and 13's auto-suspend knobs (maxIdleDurationSeconds, suspendedDurationSeconds, autoResumeEnabled) matter more here than almost anywhere else in this book, because a coding-assistant session has genuinely idle stretches — the time between the user reading a response and typing the next message, or between an autonomous agent's tool call and the model's next decision. Suspending during those gaps and resuming on the next tool call is exactly the cost lever Chapter 6 described, applied to a workload that's idle by nature rather than idle by accident.
  • Chapter 11's multi-tenant framing (one MicroVM per user vs. shared MicroVMs) becomes a per-conversation decision instead of a per-user one: does each coding session get its own dedicated instance, or do multiple sessions share one, cleaned between them? A coding assistant that lets a user run arbitrary shell commands is a strong argument for the dedicated-instance side of that tradeoff — the same reasoning Chapter 11 already laid out for any workload where state leaking between tenants would be a real security problem, not just a UX bug.
🏭 In production

A turn budget — a hard cap on how many tool calls one session gets before it must checkpoint or terminate — is a cheap first line of defense against a session that's stuck in a loop, independent of and complementary to the 8-hour wall-clock limit. The challenge below implements exactly this as ToolCallRouter.max_turns.

State Preservation for Incremental Builds

The entire premise of a coding-assistant sandbox is that "run this code" isn't a stateless operation the way it might be in Chapter 7's basic executor — the point of an incremental build is that files written in turn 3 are still there in turn 30. Chapter 3 already established that a Lambda MicroVM's disk, memory, and processes persist across a suspend/resume cycle; Chapter 9 covered the filesystem-isolation mechanics of that persistent disk in more depth. This chapter doesn't re-derive either — it's the direct beneficiary of both.

examples/ch15_repl_session.py's Turn 5 is a deliberately small proof of the same property: the file written and then rewritten in earlier turns is still readable, byte for byte, in a later turn, on the same Session object. On a real MicroVM this is the disk surviving a suspend/resume cycle exactly as Chapter 3 described; in this local simulation it's just a temp directory that happens to still exist because nothing deleted it — a much weaker guarantee, worth naming honestly rather than implying the simulation proves anything about the real service's persistence behavior.

Incremental builds specifically benefit from this in a way that's worth naming: a compiler's cache, a package manager's downloaded dependencies (Chapter 8's territory), and a test runner's coverage data all get slower or need to be rebuilt from scratch if the workspace resets between turns. A coding assistant that has to reinstall dependencies on every single tool call isn't just slow — it's fighting the exact persistence guarantee the underlying MicroVM already provides for free.

Integration with LLM Tool Calling

Everything upstream of the sandbox is an LLM API's tool-calling mechanism: the model is given a list of available tools (typically named something like write_file, read_file, run_command) with a schema for each one's arguments, and instead of returning plain text, the model's response can request a call to one of them. The caller executes that call against whatever real system backs it — here, the sandbox — and returns the result as a new message in the conversation, and the loop continues.

This chapter doesn't attempt to describe Anthropic's or any other vendor's exact tool-calling wire format, since that's the kind of implementation detail that's genuinely well-documented in each vendor's own API reference and would only go stale here. The pattern this chapter's code demonstrates is the part that stays the same regardless of which API sits on top of it: a router that takes a structured tool call, dispatches it to the right sandbox method, and returns a plain-dict result shaped for feeding straight back into a conversation.

Two things about this shape matter more than the specific method names. First, the router raises on an unrecognized tool name rather than silently no-op'ing — a model hallucinating a tool that doesn't exist is a real, common failure mode, and surfacing it as an explicit error the model can see and correct from is far better than a call that appears to succeed but did nothing. Second, the dict this returns is deliberately minimal and JSON-shaped: whatever downstream code turns this into the specific tool-result message format a given LLM API expects, it's working from plain data, not from a ToolResult dataclass instance it would need to know how to serialize.

🔬 Under the hood

Claude Code's own sandboxing docs note that "subagents run in the same process as the parent session and use the same sandbox configuration" — confirming that in that specific product, tool-calling and sandbox enforcement are handled at different layers of the same running process, not two separate systems bolted together after the fact. That's consistent with the general shape this chapter's router demonstrates: the LLM-facing tool-calling layer and the sandbox-enforcement layer are separable concerns that still need to agree on the same boundary.

Challenge: Build a Minimal AI Code Assistant

Your challenge is to implement the piece this chapter's prose described but left as an exercise: a ToolCallRouter that dispatches structured tool calls against a sandbox and enforces a turn budget for a long-running session.

Open challenges/ch15.py and implement ToolCallRouter:

  • __init__(sandbox, max_turns=20) — stores the sandbox and turn budget; starts turns_used at 0.
  • dispatch(call) — increments turns_used; raises TurnBudgetExceeded if that exceeds max_turns (before doing anything else); otherwise routes call.name to the matching sandbox method (write_file, read_file, run_command), raising ValueError for anything else, and returns a {"tool": ..., "ok": ..., "output": ...} dict.
  • remaining_turns() — returns how many turns are left in the budget.

Run the tests:

cd challenges && python3 -m pytest ch15_test.py -v

Twelve tests cover routing for each tool, an unknown-tool error, and every turn-budget edge case, including that the call which exceeds the budget is never dispatched. See solutions/ch15.py when you're ready to compare.

Run examples/ch15_repl_session.py first to see the full read-eval-print loop — write, fail, fix, succeed, read back, and a rejected path-escape attempt — against the local Session simulation, entirely offline. Then implement the challenge's ToolCallRouter to see how the same kind of sandbox plugs into an LLM's tool-calling loop.
