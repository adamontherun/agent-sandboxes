Chapter 16

Security Testing Sandbox

Chapter 15 used a MicroVM as a workspace an AI coding assistant drives turn by turn. This chapter uses the same isolation primitives for the opposite purpose: not helping code run successfully, but watching suspicious code run safely — detonating untrusted samples, scanning for vulnerabilities, observing network behavior, preserving forensic evidence, and guaranteeing that every analysis environment is torn down once the evidence is secure.

Running Scanners and Detonating Samples Safely

A vulnerability scanner or a malware-analysis detonation chamber shares the same core requirement as Chapter 7's code executor: you're running something untrusted, and you need the result without letting the thing being run damage anything outside its boundary. The difference is one of intent. A code executor runs user-submitted code that's probably benign and might occasionally be malicious by accident. A security-testing sandbox runs artifacts that are expected to be malicious, and the whole point is to observe what they try to do.

The pattern is straightforward once you have the primitives earlier chapters built: launch a dedicated MicroVM per analysis target (one sample per VM, never shared — Chapter 11's per-tenant isolation argument applies even more strongly here, since cross-contamination between samples would invalidate both analyses), feed the untrusted artifact into it, let it execute under observation, capture what happened, and tear the environment down. What changes from a normal agent sandbox is how the network and lifecycle are configured: stricter egress, more careful evidence preservation, and an even harder guarantee on teardown.

examples/ch16_malware_analysis_pipeline.py demonstrates this end-to-end pipeline. To be explicit about what it is: this is a local simulation of a MicroVM boundary, not an actual MicroVM. The AnalysisSandbox class wraps a temp directory and an in-process NetworkPolicy object — no RunMicrovm call, no Firecracker, no real VM isolation anywhere in the script. The point is to show the pipeline's shape cheaply, before wiring the same shape onto the real API calls Chapters 4-6 covered.

The "malicious sample" in this example is entirely synthetic and benign: a plain Python function that writes a marker file (simulating a payload drop) and attempts one outbound connection to 192.0.2.1 (an RFC 5737 TEST-NET-1 address that routes nowhere on the real internet), which the harness intercepts locally and logs as a blocked C2 callback attempt. No real malware appears anywhere.

Network Isolation for Malware Analysis

Chapter 7 covered egress control for a normal agent sandbox: you launch a MicroVM with the INTERNET_EGRESS network connector to give it outbound internet access, or omit it to block egress entirely. For a coding assistant (Chapter 15), some egress is usually needed — package installs, API calls, pulling dependencies. For malware analysis, the requirement flips: you almost never want real internet access, because the entire point is to observe what the sample tries to reach without letting it actually get there.

In a production malware-analysis lab, the standard approach is to pair a fully network-isolated sandbox with a fake-internet simulator — a service that answers DNS queries, serves fake HTTP responses, accepts SMTP connections, and generally pretends to be the internet from the sample's perspective while logging every request. INetSim is the canonical tool for this: it simulates common internet services in a lab environment specifically for analyzing malware network behavior, letting analysts observe C2 callback attempts, download requests, and exfiltration attempts without any traffic leaving the lab network.

On Lambda MicroVMs, the simplest version of this is launching with no INTERNET_EGRESS connector at all — the sample's outbound connections fail at the network layer, and you can observe the failure from inside the VM (or from CloudWatch logs, Chapter 13's territory). A more sophisticated setup would route traffic from the MicroVM through a monitoring proxy or an INetSim instance running in a separate MicroVM on the same private network, so you capture not just "it tried to connect" but "it tried to POST these specific bytes to this specific URL." That second pattern is beyond what a single chapter can build end-to-end, but the principle is the same as Chapter 7's egress connector: the network boundary is configured at launch time, not enforced by application code inside the VM.

🏭 In production

Real malware-analysis platforms (Cuckoo Sandbox and its successors, commercial sandboxes like Joe Sandbox or ANY.RUN) typically run the fake-internet service on a separate host from the detonation VM, connected by a virtual network bridge the analyst controls. The MicroVM equivalent would be two MicroVMs — one running the sample, one running INetSim — connected through a private network with no route to the real internet. The key constraint is that the sample VM must never share an egress path with anything that has real internet access.

The example script simulates this at the application level with a NetworkPolicy object whose attempt_connection() method always returns False (blocked) and records the attempt. On a real MicroVM the blocking would happen at the network layer instead, and the recording would come from packet capture or proxy logs rather than an in-process Python object — but the observable result from the pipeline's perspective is the same: a list of destinations the sample tried to reach, all blocked, all logged.

Forensic Snapshot Preservation

The worst thing you can do after a suspicious sample runs is tear down the environment immediately. A terminated MicroVM's disk contents are gone — TerminateMicrovm is final (Chapter 6 covered this). If the sample wrote files, spawned processes, or left any other artifact behind, you need to capture that evidence before termination.

The pattern is: once analysis completes (or once a suspicious signal fires — a blocked network attempt, an unexpected file write, a process that shouldn't exist), suspend first, export second, terminate last. Chapter 3 and 6 established that a suspended MicroVM preserves its full state — memory, disk, running processes. That gives you a stable snapshot to export from without racing against a live, possibly still-mutating environment.

The SHA-256 hash over exported artifacts is the simplest form of chain-of-custody integrity: if you later need to prove the evidence hasn't been tampered with (or just confirm that the export captured what you think it captured), the hash computed at export time is your reference. In a real forensics workflow you'd also record a timestamp, the analyst's identity, and store the hash in a tamper-evident log separate from the evidence itself — but the principle starts here: hash it at the moment you export it, before anything else touches it.

🔬 Under the hood

Chapter 9's filesystem isolation means the artifacts you're exporting are already scoped to the MicroVM's own disk — there's no risk of accidentally exporting host-system files alongside the sample's output, because the sample never had access to host-system files in the first place. The isolation boundary that protected the host during execution also protects the evidence's integrity during export: what you get out is exactly and only what the sample put in.

Automated Teardown After Analysis

A MicroVM running a vulnerability scanner or a detonation chamber is itself a liability the moment analysis is done. It contains the artifact you were analyzing, possibly in a partially-executed state, with whatever filesystem and memory modifications the sample made still live. Leaving it running — or even suspended — after evidence is preserved is risk with no upside.

The guarantee this chapter's pipeline pattern enforces is simple: TerminateMicrovm always runs once evidence is preserved, even if something in the analysis or evidence-preservation phase raised an exception. This is the same structural guarantee Chapter 14's production orchestrator used for error recovery (always clean up, even on failure), applied to a context where "failure to clean up" isn't just a resource leak — it's a live malware-analysis environment sitting unattended.

The finally block is the critical structural choice. Whether the sample ran cleanly, whether it raised an exception, whether evidence preservation failed halfway through — the sandbox gets terminated regardless. On a real Lambda MicroVM deployment this would be a TerminateMicrovm call in the same position, with the same guarantee: no code path through this function exits without terminating the analysis environment.

⚠️ Gotcha

If evidence preservation itself raises (disk full, permission error, network timeout copying to remote storage), you lose the evidence but you still need the teardown. Don't put the TerminateMicrovm call after the evidence export in a linear sequence — put it in a finally block that runs regardless. The alternative — a lingering analysis MicroVM that outlives a failed export — is strictly worse than lost evidence, because it's lost evidence plus a live liability. Chapter 10's resource-limits discussion applies here too: the MicroVM's 8-hour maximumDurationInSeconds ceiling is a backstop, not a cleanup strategy.

Challenge: Build a Malware Analysis Pipeline

Your challenge is to implement the core of the pipeline this chapter described: an AnalysisPipeline class that runs a sample function under a network policy, captures file writes and blocked connection attempts, computes an integrity hash over the captured artifacts, and guarantees a teardown callback fires even if the sample raises.

Open challenges/ch16.py and implement:

  • NetworkPolicy.attempt_connection(destination) — records a NetworkAttempt (blocked=True) and returns False.
  • AnalysisPipeline.__init__(sample_id, teardown_callback) — stores both, initializes self.network, self.files (dict), and self._torn_down (False).
  • AnalysisPipeline.write_file(name, content) — records content (bytes) under name in self.files.
  • AnalysisPipeline.run(sample_fn) — calls sample_fn(self) in a try/finally that always calls teardown_callback() and sets _torn_down = True; on success, returns an Evidence dataclass with artifact_hash computed as the SHA-256 of all file contents concatenated in sorted filename order.
  • AnalysisPipeline.compute_hash() — the standalone hash computation (SHA-256 of concatenated file contents in sorted key order; empty-bytes hash if no files).

Run the tests:

cd challenges && python3 -m pytest ch16_test.py -v

Fifteen tests cover the network policy, file recording, hash computation (including sorted-order correctness), successful runs, teardown on exceptions, and evidence integrity. See solutions/ch16.py when you're ready to compare.
